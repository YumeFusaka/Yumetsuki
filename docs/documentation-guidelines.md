# 文档规范

## 语言规则

- 项目文档默认使用中文撰写
- 路径、类名、函数名、模块名、命令、配置键名和 git commit message 可保留英文
- 不建议中英文叙述风格混用

## 层级规则

- 路线图文档：定义阶段目标、范围与依赖
- 设计 spec：定义专题方案、边界、风险、默认策略
- 实施 plan：定义任务拆分、测试步骤与提交节奏

## 参数规则

- 设计阶段中的关键数值默认应配置化
- 包括但不限于：
  - 超时
  - 并发数
  - 缓冲长度
  - 滑动窗口大小
  - prompt 预算
  - 刷新频率

除非存在强约束，不建议在 spec 中永久写死这些值。

## 配置化原则

- 凡是显著影响桌宠体验、响应节奏、记忆强度、TTS 稳定性和代理行为的关键参数，默认都应配置化
- 是否直接暴露给普通用户，是第二层决策；第一层要求始终是“先可配置，再决定是否开放 UI”
- 不允许把关键体验参数以黑盒形式散落硬编码在实现中
- 当前仓库中已存在的历史硬编码参数，后续在对应模块重构或增强时，应优先迁移到配置层

### 应优先配置化的参数类型

- 短期记忆相关：
  - recent turns 窗口大小
  - working facts 上限
  - facts 衰减速度
  - active tasks 保留策略
  - 热上下文预算
- TTS 相关：
  - 句段并发数
  - 读超时
  - 单句段总超时
  - 队列长度
  - 回退策略
- UI / 桌宠体验相关：
  - 字体
  - 字号
  - 被动互动频率
  - 气泡停留时长
  - 动画与显示节奏
- Agent / 工具相关：
  - 路由阈值
  - 多步上限
  - 工具超时
  - OCR 频率
  - 浏览器等待时间

### 文档书写要求

- spec 中出现具体数字时，默认只能作为“建议默认值”或“示例值”出现，不能暗示该值会被永久硬编码进实现
- plan 中若给出示例代码，涉及上述关键参数时，应优先写成“从配置读取”的形式，而不是直接写死字面量
