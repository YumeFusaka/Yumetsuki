# 后续阶段功能头脑风暴记录

> 日期：2026-05-27 至 2026-05-28
>
> 状态：历史参考。第一轮 5 agent 独立构思和第二轮交叉评审已完成，核心结论已吸收到 [Phase 7 诊断能力实施计划](../plans/2026-05-28-phase-7-diagnostics.md) 与 [Phase 8 / Phase 9 阶段设计草案](../specs/2026-05-28-phase-8-9-design.md)。

## 背景

本记录整理一次面向 Yumetsuki 后续阶段的 5 agent 头脑风暴。参与视角分别为：

- 产品体验与桌宠交互
- Agent 智能、记忆与长期陪伴
- 插件生态、MCP 与自动化能力
- 多模态、语音、视觉与角色演出
- 工程稳定性、性能、测试与发布维护

共同前提：

- 第六阶段实现、本地自动化验证和 sub agent 复审已完成。
- 真实浏览器、OCR、MCP、STT / TTS / API 实机验收仍暂缓。
- 后续扩展应继续遵守原版兼容优先、显式扩展触发、本地敏感配置不提交、改动可本地验证等项目规则。
- 本轮只做功能构思和优先级判断，不代表已确认实施计划。

## 汇总结论

五个 agent 的共识很明确：下一阶段不宜继续优先堆新大能力，而应先把已经存在的工具、语音、视觉、日志、记忆和主动行为做成更可见、可控、可诊断、可回溯的产品闭环。

综合两轮建议，最值得进入下一阶段讨论的方向是：

1. **实机验收与诊断体系**
   - 包含真实 API、TTS、STT、OCR、MCP、Playwright Edge、麦克风、模型目录等分项诊断。
   - 目标是把当前“真实联调暂缓”的风险转成可重复执行、可记录、可排障的流程。
2. **平台日志排障增强与工具执行审计**
   - 平台日志从“能看日志”升级为“能定位问题”。
   - 工具调用需要单独记录来源、权限、参数摘要、耗时、结果、失败原因和关联会话。
3. **记忆质量治理与会话连续性**
   - 长期记忆写入前增加候选、来源、置信度、去重、冲突和升格原因。
   - 更充分利用 `SessionContext` 中的会话摘要、情绪状态、关系状态和当前主题。
4. **聊天窗与语音对话产品闭环**
   - 强化停止、重试、重播、复制、追问、失败恢复和语音轮次状态反馈。
   - 先做状态机和取消语义，不急于引入真正流式 STT。
5. **轻量日常陪伴与主动行为 2.0**
   - 基于上下文、空闲时长、当前主题、情绪状态和冷却策略触发低打扰陪伴。
   - 不做重型日程系统，不做强提醒软件替代品。

## 候选方案清单

### 1. 实机验收与诊断向导

来源视角：工程稳定性、多模态。

目标：

- 在设置中心增加“诊断 / 验收”入口。
- 分项检查 API、TTS、STT、OCR、MCP、Playwright Edge、麦克风和模型目录。
- 每项输出状态、耗时、错误类型和建议处理，并写入平台日志。

用户价值：

- 用户不用在聊天链路随机失败后猜原因。
- 后续恢复实机验收时有标准入口和标准记录。

依赖：

- `LogService`
- `STTManager`
- TTS adapter
- `VisionManager`
- `MCPHost`
- `web_automation`
- 设置中心 UI

主要风险：

- 真实设备和外部服务状态不可控。
- 测试不能泄露 API key、私有 URL、截图、模型路径全文等敏感信息。
- 慢服务必须有超时和取消。

验证方式：

- mock 各后端成功、失败、超时。
- Qt offscreen 实例化诊断页。
- 手工实机矩阵覆盖 API、TTS、STT、OCR、MCP、Playwright。

第一轮评分：

- 多模态视角：20 / 20
- 工程稳定性视角：19 / 20

### 2. 平台日志排障增强与一键诊断包

来源视角：工程稳定性。

目标：

- 日志事件增加 `trace_id`、`request_id`、`session_id`、`stage` 等字段，用于串联一次用户请求。
- 平台日志详情区显示可能原因和建议操作。
- 支持导出脱敏诊断包，包含脱敏日志、版本信息、平台信息、依赖探测结果和非敏感配置摘要。

用户价值：

- 用户或开发者可以用同一份诊断包排查 TTS 没声音、STT 卡住、MCP 连不上、OCR 失败等问题。

依赖：

- `core/log_service.py`
- `core/log_sanitizer.py`
- 平台日志页面
- 已有结构化日志事件

主要风险：

- 脱敏必须可靠。
- 不应导出截图原图、API key、本地隐私文件内容、私有服务地址全文。
- 诊断包体积需要限制。

验证方式：

- 脱敏单测。
- 导出内容快照测试。
- 平台日志筛选、复制、导出回归。

第一轮评分：

- 工程稳定性视角：20 / 20

### 3. 工具执行审计与回放面板

来源视角：插件生态、MCP 与自动化。

目标：

- 记录本地插件工具和 MCP 工具调用的来源、工具名、权限等级、参数摘要、耗时、结果摘要、失败原因和关联会话。
- 可在平台日志内增加工具工作区，或后续独立成“工具日志 / 自动化日志”视图。

用户价值：

- 用户能看清 AI 到底调用了什么工具。
- 高权限能力如系统命令、浏览器点击 / 填写、MCP 调用具备可追踪基础。

依赖：

- `ToolRegistry`
- `LogService`
- 插件 / MCP 来源名称
- 平台日志 UI

主要风险：

- 工具参数可能包含 URL、路径、网页文本、MCP 返回数据，需要脱敏和摘要化。

验证方式：

- mock 插件工具成功、失败、超时。
- mock MCP `tools/call`。
- 验证日志脱敏、筛选、导出、UI 刷新不阻塞。

第一轮评分：

- 插件生态视角：20 / 20

### 4. 配置健康检查与安全治理

来源视角：工程稳定性。

目标：

- 统一检查 `api.yaml`、`system_config.yaml`、`mcp.yaml`、`memory.yaml`、`agent.yaml` 的可用性、兼容性和风险项。
- API 页、系统页、MCP 页保存后可以显示局部检查结果。
- 对 TTS 模式给出稳定性提示，例如 `wav + inline` 是当前保底组合，`pcm_stream + auto` 不应标记为稳定方案。

用户价值：

- 保存配置时即可发现模型路径无效、超时不合理、MCP server 配置错误、TTS 模式组合不稳定等问题。

依赖：

- `config/schema.py`
- `config/manager.py`
- 设置页保存语义
- TTS / STT / OCR 文档化边界

主要风险：

- 不能把高级用户配置误报成错误。
- 第一版应只做高价值 warning / error / info，不做自动修复。

验证方式：

- 参数化单测覆盖缺失、无效、过期字段、危险组合。
- 设置页保存失败回滚语义回归。
- 脱敏测试，确保不输出敏感配置明文。

第一轮评分：

- 工程稳定性视角：19 / 20

### 5. 记忆质量治理与可回溯记忆账本

来源视角：Agent 智能、记忆与长期陪伴。

目标：

- 在写入 Mem0 前增加记忆候选结构，记录来源 turn、类型、置信度、升格原因。
- 增强去重、冲突判断、短期约束 TTL 检查。
- 对话日志显示本轮新增 / 引用记忆；平台日志记录记忆升格、跳过、冲突。

用户价值：

- 降低“记错”“重复记”“把临时约束当长期偏好”的概率。
- 用户更容易信任长期陪伴能力。

依赖：

- `SessionPolicy.collect_mem0_candidates`
- `Mem0MemoryStore.add_memory`
- `LogService`
- 对话日志页 memory tag

主要风险：

- 过度治理导致该记的不记。
- 第一版不应做复杂自动删除，只处理明显重复、短期约束和低置信度内容。

验证方式：

- 重复候选跳过。
- 短期约束不升格。
- 显式“记住”优先升格。
- 日志记录原因。
- Mem0 mock 写入次数。

第一轮评分：

- Agent / 记忆视角：19 / 20

### 6. 会话状态画像与关系连续性

来源视角：Agent 智能、记忆与长期陪伴。

目标：

- 让 `SessionSummary.mood_state`、`relationship_state`、`current_topic` 更充分参与对话。
- 回复后异步更新会话摘要、用户情绪、关系状态；热路径只读取已有摘要，不在首字前增加额外 LLM 调用。

用户价值：

- 角色回复更连贯。
- 被动发言、情绪标签和长期陪伴感更自然。

依赖：

- `SessionContext.summary`
- `SessionPolicy`
- `AgentReflector`
- `TextProcessor`
- 平台日志

主要风险：

- 用户情绪或关系状态误判会导致角色反应过度。
- 需要低置信度丢弃、最近状态优先、日志解释来源。

验证方式：

- 摘要更新。
- prompt 注入。
- 低置信度丢弃。
- 不阻塞首字响应。
- 日志包含状态变化事件。

第一轮评分：

- Agent / 记忆视角：18 / 20

### 7. 聊天窗微交互增强

来源视角：产品体验与桌宠交互。

目标：

- 补齐回复中断后的状态恢复、失败后的重试体验、上一轮追问入口、最后一句重播 TTS、复制角色回复、被动气泡点击展开上下文等小闭环。

用户价值：

- 用户更容易理解桌宠当前状态。
- 长回复、失败、误触语音、TTS 播放问题都有恢复路径。

依赖：

- `ChatWindow`
- TTS pipeline
- 对话日志
- 平台日志
- `SessionContext`

主要风险：

- 聊天窗 UI 已经复杂，新增操作容易拥挤。
- 需要保持发送按钮、停止语义、状态条和输入框之间的职责清楚。

验证方式：

- Qt offscreen 实例化。
- 忙碌态停止。
- 失败重试。
- TTS 重播。
- 被动气泡点击恢复。
- 长文本滚动不被破坏。

第一轮评分：

- 产品体验视角：20 / 20

### 8. 日常陪伴循环与主动行为 2.0

来源视角：产品体验、Agent 智能。

目标：

- 把主动行为从“闲置后说一句话”升级为低打扰日常陪伴闭环。
- 支持久坐提醒、休息提醒、喝水提醒、睡前问候、工作陪伴等轻量事件。
- 主动发言参考最近主题、情绪状态、未完成任务、互动节奏和角色上下文。

用户价值：

- 桌宠更像陪伴对象，而不是聊天入口。
- 主动开口能体现“记得刚才发生了什么”。

依赖：

- `ProactiveScheduler`
- 被动气泡
- `SessionContextManager`
- 情绪标签
- TTS
- 平台日志

主要风险：

- 过频会打扰用户。
- 需要继续保留全局冷却、活跃时段、被动状态约束。
- 不应演变成完整日程软件或强提醒系统。

验证方式：

- mock 时间和上下文。
- 验证触发理由、冷却、prompt 内容、被动气泡显示、点击进入主对话、TTS 入队和日志事件。

第一轮评分：

- 产品体验视角：18 / 20
- Agent / 记忆视角：17 / 20

### 9. 低延迟语音对话闭环

来源视角：多模态、语音、视觉与角色演出。

目标：

- 优化“录音 -> STT -> Agent -> TTS”的语音轮次状态反馈和取消语义。
- 增加识别中、思考中、说话中等细分状态。

用户价值：

依赖：

- TTS pipeline

- 平台日志

主要风险：

- 真正流式 STT 牵涉较大，近期不建议直接做。
- TTS PCM 分支仍有服务端兼容风险。
- 保底路径应继续使用 `wav + inline`。

验证方式：

- 状态机单测覆盖录音、识别、生成、播报、取消。
- 实机记录端到端耗时。
- 低延迟路径作为显式增强，不替代保底路径。

第一轮评分：

- 多模态视角：18 / 20

用户价值：

验证方式：

- 情绪解析。
- 默认回退。
- 资源缺失回退。

第一轮评分：

- 多模态视角：18 / 20

- 新模板插件加载测试。

## 暂不建议近期优先做

以下方向有价值，但本轮 agent 普遍认为不适合立即进入下一阶段主线：

- **后台持续看屏 / 自动视觉监控**
  - 隐私风险高，且当前视觉输入明确坚持显式触发。
- **真正流式 STT 边说边识别**
  - 当前 STT 适配固定为 WAV 转写，直接改成流式会牵动录音、转写、取消、状态条和 Agent 触发时机。
- **复杂图像语义理解 / 多模态 VLM**
  - 当前视觉范围是 OCR 文本，不做复杂图像语义推理。
- **完整外部插件 marketplace / 签名生态**
  - 维护成本高，安全承诺容易过度。近期更适合先做导入校验、风险提示和元数据规范。
- **全自动无人值守网页 / 桌面操作**
  - 当前应先补审计、确认、取消和失败恢复。
- **任意系统命令自动规划执行**
  - `system_control.run_command` 是 high 权限能力，应保持显式、受限、可追踪。
- **多步任务状态持久化与恢复**
  - 价值明确，但会混入浏览器、MCP、OCR 等未实机稳定链路的问题，建议实机验收之后再做。
- **重型人格成长系统**
  - 长期等级、亲密度数值和复杂情绪模型应建立在可靠记忆和会话摘要之上。
- **打包发布与兼容性矩阵**
  - 实机验收尚未恢复前直接打包，容易把未确认问题转移到发布层。

## 推荐阶段拆分（第一轮候选）

本节保留第一轮阶段拆分口径，作为历史参考。当前推荐口径以本文后段“第二轮后的推荐阶段”为准。

### Phase 7 候选：稳定性与可诊断性收口

优先目标：

- 实机验收与诊断向导
- 平台日志排障增强与一键诊断包
- 工具执行审计
- 配置健康检查

理由：

- 直接对应当前项目最大未收口风险。
- 这些能力是后续自动化、语音、多模态、记忆成长继续扩展前的基础设施。
- 大多能通过 mock 自动化测试和少量手工实机矩阵验证。

### Phase 8 候选：陪伴连续性与交互闭环

优先目标：

- 记忆质量治理与可回溯记忆账本
- 会话状态画像与关系连续性
- 聊天窗微交互增强
- 日常陪伴循环 / 主动行为 2.0
- 低延迟语音对话闭环的状态机部分

理由：

- 在诊断和可观测性打稳后，再增强陪伴感和交互体验，失败时更容易定位。
- 这些能力复用现有 `SessionContext`、主动行为、被动气泡、TTS、平台日志和对话日志。

### Phase 9 候选：生态、Computer Use 与高级体验

优先目标：

- 插件 SDK 开发者体验增强
- MCP 兼容性实机模板
- Computer Use / 桌面应用自动化 MVP
- 显式读屏区域选择
- 情绪驱动演出调度器

理由：

- 这些方向会放大工具、浏览器、桌面应用、视觉和角色演出的能力边界。
- 适合在审计、诊断、配置健康和基础实机验收完成后推进。

## 第二轮交叉评审

第二轮已在 2026-05-28 完成。5 个 agent 基于第一轮去重清单互相评审，分别从产品体验、Agent / 记忆、插件 / MCP、多模态和工程稳定性视角重新排序、指出反对意见、标记依赖关系并给出阶段拆分。

第二轮的核心结论比第一轮更收敛：

- Phase 7 不应继续新增大型能力，应先把现有能力做成可诊断、可复现、可脱敏导出、可追踪。
- 日志 trace、诊断包、配置健康、工具审计和实机诊断是后续语音、OCR、MCP、浏览器、主动行为和 Computer Use / 桌面应用自动化的前置基础。
- 产品侧提出的聊天窗失败恢复不应被推太远；Phase 7 可以纳入最小恢复闭环，但完整微交互增强放到 Phase 8。
- 记忆质量治理、会话画像和主动陪伴应建立在 trace 与可解释日志之上，避免把不可追溯的记忆和情绪判断扩散到长期陪伴。
- 低延迟语音近期只做状态机、取消语义和耗时指标，不做真正流式 STT，不把 PCM / 会话扩展当作稳定默认路径。
- 插件 SDK、MCP 模板、区域读屏、情绪演出和 Computer Use / 桌面应用自动化都应后置到诊断、审计和实机基础稳定之后。

- “有效均分”基于第二轮各 agent 对该项给出的总分计算；未评分视角不计入均分。
- 最终优先级不只看分数，还考虑依赖关系、风险放大效应和当前项目短板。

### 第二轮反对意见与降级项

#### 工具审计不要一开始做成重型独立回放系统

插件 / MCP 视角认为工具审计是 Phase 7 硬前置；产品和多模态视角则提醒不要过早新建复杂独立页面。最终折中：

- Phase 7 先接入平台日志的“工具工作区”或工具筛选视图。
- 记录来源、工具名、权限、参数摘要、结果摘要、耗时、失败原因、关联会话和 trace id。
- “回放”第一版只支持回看，不自动重放高权限操作。
- 独立工具日志页面、自动回放和重试放到后续阶段。

#### 聊天窗微交互要做，但不能塞满主界面

产品视角把聊天窗微交互排第一；工程和 Agent 视角认为它不能压过诊断主线。最终折中：

- Phase 7 做最小恢复闭环：失败后保留输入、显示打开平台日志入口、提供明确重试 / 停止状态。
- Phase 8 再扩展最近一轮操作区、最后一句 TTS 重播、复制角色回复、被动气泡点击展开上下文等完整体验。
- 操作优先放在状态条、hover、右键菜单或最近一轮操作区，避免把聊天窗做成工具面板。

#### 配置健康检查应并入诊断体系

多个 agent 都给配置健康高分，但产品侧提醒不要新增一个用户难理解的系统。最终折中：

- 第一版作为诊断向导和设置页保存后的局部检查能力。
- 输出 warning / error / info。
- 不自动改写用户配置。
- 对高级用户配置保持宽容，不把实验路径误报为阻塞错误。

#### 低延迟语音只做状态机版

多模态和工程稳定性视角都反对近期直接做真正流式 STT。最终范围：

- 只做录音中、识别中、思考中、说话中、停止中、失败可重试等状态。
- 记录录音结束到 STT 完成、首字、首音频的耗时指标。
- 继续把 `wav + inline` 作为稳定保底组合。
- PCM / `auto` / `session_preload` 仍作为显式扩展或实验路径，不作为默认稳定方案。

#### 主动行为和情绪演出后置

主动行为 2.0、日常陪伴、情绪演出都被认可，但不适合作为 Phase 7 主线：

- 主动行为必须先有可靠的会话画像、记忆账本、冷却策略、触发理由日志和 TTS 状态。
- 情绪演出调度必须先保证 `[emotion:...] -> 立绘切换` 的现有链路稳定，再考虑气泡停留、TTS 风格等可选映射。
- 服务端情绪参数只能作为显式可选扩展，不能破坏原版 TTS 路径。

#### Computer Use / 桌面应用自动化必须后置

用户后续补充明确指出，目标不是“打开软件、调音量”等单点工具，而是让 Yumetsuki 真正具备使用电脑应用的 Computer Use 能力。这个方向价值明确，但必须后置：

- 工具审计先于桌面应用自动化。
- 用户确认、失败中止、取消恢复和权限拦截先于多步自动化。
- 浏览器、MCP、OCR、Windows UI Automation、截图 / OCR 观察和系统动作等实机链路稳定后，再考虑 Computer Use MVP。
- 第一版应以窗口枚举 / 聚焦、快捷键、剪贴板、媒体键、应用专用接口和 UI Automation 控件操作为主；坐标点击只能作为显式确认后的兜底。
- 不承诺任意应用 100% 稳定可控，不做无人值守高权限桌面操作。

### 第二轮依赖关系

强依赖链如下：

```text
日志 trace / request_id / session_id / stage
→ 平台日志诊断包
→ 工具审计、Agent 决策摘要、记忆账本、实机诊断落点
```

```text
配置健康检查
→ 实机验收向导
→ 低延迟语音、OCR 区域选择、MCP 模板、发布打包
```

```text
工具执行审计
→ Computer Use / 桌面应用自动化
→ 主动行为触发工具、多步任务恢复、需用户确认的应用操作
```

```text
记忆质量治理
→ 会话状态画像
→ 主动行为 2.0
→ 情绪驱动演出调度器 / 人格成长类能力
```

```text
实机验收与诊断向导
→ 低延迟语音状态机
→ TTS 重播 / 停止说话 / 语音陪伴体验
```

### 第二轮后的推荐阶段

#### Phase 7：可诊断性与最小恢复闭环

目标：不新增大能力，先让现有能力的问题可定位、可复现、可脱敏导出、可从聊天窗恢复。

实施计划：[Phase 7 诊断能力实施计划](../plans/2026-05-28-phase-7-diagnostics.md)

优先目标：

- 平台日志 trace / request_id / session_id / stage 增强
- 一键脱敏诊断包
- 实机验收与诊断向导第一版
- 配置健康检查第一版
- 工具执行审计第一版，先并入平台日志
- 聊天窗最小失败恢复：保留输入、重试、停止状态、打开平台日志入口

验收重点：

- mock 自动化测试覆盖成功、失败、超时、取消。
- 诊断包脱敏快照通过。
- 本机手工矩阵能逐项记录 API、TTS、STT、OCR、MCP、Playwright 结果。
- 工具高权限调用有来源、权限、参数摘要和结果摘要。
- 不导出截图原图、音频、API key、私有 URL 全文、模型路径全文或 OCR 原文长文本。

#### Phase 8：记忆可信度、交互闭环与低打扰陪伴

目标：在可观测性打稳后，提高桌宠陪伴连续性和日常可用性。

设计草案：[Phase 8 / Phase 9 阶段设计草案](../specs/2026-05-28-phase-8-9-design.md)

优先目标：

- 记忆质量治理与可回溯记忆账本
- 会话状态画像与关系连续性，异步更新、热路径只读缓存
- 聊天窗完整微交互增强：最近一轮操作区、复制、重播、失败恢复、被动气泡展开
- 低延迟语音对话闭环的状态机、取消语义和耗时指标
- 日常陪伴循环 / 主动行为 2.0 的低打扰版本

验收重点：

- 记忆候选有来源、置信度、升格原因、冲突和跳过原因。
- 画像更新不增加首字前 LLM 调用。
- 主动行为有触发理由、冷却、活跃时段和低打扰约束。
- 语音状态机可取消、可恢复、可记录耗时。

#### Phase 9：生态、Computer Use、视觉与演出扩展

目标：在诊断、审计、记忆和基础交互稳定后，扩展生态、桌面应用使用能力与高级体验。

路线图草案：[Phase 8 / Phase 9 阶段设计草案](../specs/2026-05-28-phase-8-9-design.md)

优先目标：

- 插件 SDK 开发者体验增强
- MCP 兼容性实机模板
- 显式读屏区域选择与视觉摘要
- 情绪驱动演出调度器第一版
- Computer Use / 桌面应用自动化 MVP，只读观察优先，高风险动作确认执行
- 发布打包与兼容性矩阵，放在 Phase 9 后半，并以前两阶段实机结果为输入

验收重点：

- 旧插件兼容，新模板插件可运行。
- MCP stdio / SSE / HTTP 错误分类清晰。
- 区域读屏继续坚持显式触发，不做后台看屏。
- 情绪演出有资源缺失回退，TTS 风格参数为显式可选扩展。
- Computer Use 高风险步骤必须确认，失败可中止，取消可恢复，所有步骤有审计记录。
- Computer Use 必须沿用 Phase 9 设计草案中的硬禁止清单、L1-L5 能力分层、现有插件权限映射、运行时隐私规则和 L4 / L5 限制。
