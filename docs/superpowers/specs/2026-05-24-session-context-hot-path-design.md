# SessionContext 与首字热路径设计

日期：2026-05-24

> 状态：设计已确认，可进入实施计划。

## 背景

Yumetsuki 当前已经具备：

- 角色系统
- 长期记忆（Mem0）
- Agent 路由、工具调用与反思
- 流式对话与句级 TTS

但当前聊天主路径仍有两个核心问题：

1. 缺少真正的单会话短期记忆，导致“上一句刚说完，下一句就像不知道”
2. 首字热路径承担了过多职责，首字延迟不稳定

这两个问题相互耦合：

- 如果只依赖原始 `_history`，上下文虽然存在，但缺少结构和优先级
- 如果只依赖长期记忆检索，当前会话连续性仍然不够强，且会增加首字前成本

因此需要新增一个独立的 `SessionContext` 层，把当前会话的工作记忆从长期记忆与原始历史中拆出来，并明确首字热路径只依赖极少量同步可得的状态。

## 目标

- 引入 `SessionContext` 作为单会话工作记忆主存
- 明确 `SessionContext` 与 `mem0` 的职责边界
- 让首字热路径依赖一个稳定、轻量、结构化的上下文块
- 保持桌宠多轮连续性，同时不把所有能力都塞进首字前关键路径

## 非目标

- 不在本设计中实现 STT、OCR、视觉与浏览器高级操控
- 不在本设计中讨论完整 UI 重构
- 不要求 `SessionContext` 取代长期记忆
- 不要求首字前运行额外外部 LLM 来“智能整理”短期记忆

## 设计原则

- 短期记忆是一等公民，不是 `_history` 的附属品
- `SessionContext` 只管理当前会话工作记忆，不保存长期事实全集
- 首字前只允许轻量同步更新，不允许为了“更聪明”增加额外外部 LLM 调用
- 长期记忆保留在 `mem0`
- 会话态必须支持衰减、压缩和清理，不能无限增长

## 已确定的默认策略

本设计不再保留分叉，默认采用以下策略：

- `SessionContext` 采用增强型短期记忆，而不是极薄缓存
- 程序恢复时只恢复最近会话，不在 Phase 4 引入多会话恢复管理
- 工具结果只保留对后续几轮仍有影响的关键结论，不保留大段原始输出
- `mem0` 采用保守升格策略，只吸收稳定偏好、长期约束和跨会话事实
- 首字策略以连续性优先，不为了极限首字而牺牲当前会话承接能力

## 核心结构

### `SessionContext`

`SessionContext` 是每个聊天会话的工作记忆对象，建议至少包含：

- `recent_turns`
  - 最近多轮原始对话窗口
- `working_facts`
  - 当前会话高权重临时事实
- `active_tasks`
  - 当前进行中的任务状态
- `summary`
  - 当前主题、当前会话摘要、当前氛围

### `SessionContext` 应该放什么

必须放：

- 最近多轮 user / assistant 原文
- 用户刚给出的约束、纠正、当前状态、临时决定
- 当前任务目标、任务状态、最近工具结果
- 当前主题与本轮前仍有价值的摘要

可以放，但需谨慎控制：

- 当前互动模式或轻量情绪模式
- 最近工具结果的结论性摘要

### `SessionContext` 不该放什么

不应放：

- 长期偏好全集
- 跨会话事实全集
- 大段 thinking / 内部推理文本
- 完整网页全文、完整 OCR 大文本、完整 JSON 工具结果
- 永不清理的旧情绪或旧任务状态

一句话边界：

> `SessionContext` 只保留“当前会话中，对后续几轮仍有影响，且用户忘了会明显出戏”的状态。

## 分层存储策略

### 主存

短期记忆主存应为进程内 `SessionContext` 对象。

原因：

- 读写最快
- 最适合每轮更新
- 最适合作为首字热路径直接读取的数据源

### 快照存储

短期记忆应使用 SQLite 做轻量快照，而不是直接依赖 mem0。

用途：

- 程序重启后恢复当前会话
- 为后续调试和可视化提供可查数据

### 长期沉淀

`mem0` 只负责长期记忆：

- 稳定偏好
- 跨会话事实
- 长期关系变化
- 多次重复出现的高价值模式

短期内容只有在满足升格条件时才写入 `mem0`。

## 首字热路径

### 首字前必须同步完成

只允许做以下工作：

1. 将当前用户输入加入 `recent_turns`
2. 提取显式高权重工作事实
3. 更新任务入口状态
4. 构建热上下文块
5. 主模型开始流式输出

说明：

- 这里的窗口大小、working facts 上限、任务保留条数、热上下文预算等关键数值，后续均应做成可配置项，而不是在实现中永久写死。

### 首字前不应该做

不应在首字前做：

- 深记忆检索
- 复杂摘要重算
- 多步规划
- 额外外部 LLM 调用来整理短期记忆
- TTS 翻译与播放调度

### 流式期间可轻量更新

- assistant 暂存输出更新
- 工具结果即时写入任务状态和 working facts
- 已有 facts 的 `last_seen` 刷新

### 回合结束后异步整理

- 更新 / 压缩 `summary`
- 裁切 `recent_turns`
- 衰减与清理 `working_facts`
- 收尾 `active_tasks`
- 决定哪些内容升格到 `mem0`

## 首字热上下文块

热上下文建议固定成四段：

1. 当前主题
2. 当前会话关键点
3. 当前任务状态
4. 最近对话摘录

建议格式：

```text
当前会话短期上下文:
- 当前主题: 正在讨论首字热路径的上下文组织
- 当前会话关键点:
  - 当前只讨论，不做代码工作
  - 用户认可 Phase 4/5/6 的路线划分
  - 用户希望短期记忆更强，并接受滑动窗口、压缩摘要、重要性权重
- 当前任务状态:
  - 任务: 细化 Phase 4 设计
  - 状态: 进行中
  - 当前阶段: 讨论 SessionContext 与热路径
- 最近对话:
  - 用户: 好，继续
```

### 组织原则

- 使用状态说明，而不是叙事段落
- 优先列表化呈现
- 热上下文总预算应严格受控
- 短期上下文优先级高于长期记忆补充
- 热上下文预算、最近对话摘录条数、关键点数量都应配置化

## 与 prompt 的关系

推荐层次顺序：

1. 角色核心提示
2. `SessionContext` 热上下文
3. 长期记忆补充
4. 本轮运行态补充（例如工具结果）
5. 当前用户输入

### 角色提示的边界

角色资料应拆分为：

- 热路径角色提示
  - 只保留必须的身份、称呼、语言、边界、语气基调
- 冷路径角色资料
  - 更详细的人设、剧情、资源文档

首字热路径只应使用前者。

## 与长期记忆的协作

### 短期记忆职责

- 解决“我们刚刚在聊什么”
- 解决“当前任务做到哪一步”
- 解决“刚才的纠正和约束还记得吗”

### 长期记忆职责

- 解决“这个用户长期偏好什么”
- 解决“跨会话仍然成立的事实是什么”

### 升格条件

短期内容进入 `mem0` 前应满足至少之一：

- 多次重复出现
- 用户明确要求记住
- 超过当前会话仍然成立
- 明显属于稳定偏好 / 长期事实

说明：

- 具体升格阈值、重复出现次数、importance 下限等关键数值不在设计中写死，后续实现阶段应进入配置层。

## 风险

### 风险 1：把 `SessionContext` 做成第二个 raw history

如果缺少：

- 高权重 facts
- 任务状态
- 摘要
- 衰减 / 遗忘机制

那么它只会变成更厚的原始消息堆。

### 风险 2：把太多整理工作放进首字前

如果首字前需要：

- 深记忆检索
- 外部 LLM 辅助整理
- 大量摘要重算

那么短期记忆会改善连续性，但首字会继续变慢。

### 风险 3：与长期记忆重复污染

如果短期记忆与长期记忆无明确分层：

- prompt 会重复
- 记忆会脏
- 模型会错误放大某些信息的重要性

## 验收标准

- 当前会话多轮连续性明显提升
- 用户刚刚给出的约束 / 纠正可以稳定持续若干轮
- 工具结果和任务状态不会下一轮就丢
- 首字热路径不依赖额外外部 LLM
- 长短期记忆职责边界清晰
