# Phase 6：浏览器、视觉与插件生态设计

日期：2026-05-24

> 状态：设计已确认，等待 Phase 4、Phase 5 收敛后进入详细实施。

## 背景

Phase 6 是 Yumetsuki 从“会聊天的桌宠”向“更完整的桌面代理”迈进的一步。

当前项目已经有：

- 插件系统
- MCP 接入
- 基础浏览器相关工具
- Web 自动化插件

但仍缺少：

- 更自由的浏览器持续操控
- 更清晰的插件 / MCP 管理界面
- 屏幕 OCR
- 基础视觉输入

## 目标

- 提升插件与 MCP 管理能力
- 把浏览器能力从“单次动作”提升到“持续操控”
- 增加 OCR 与屏幕理解
- 让视觉输入进入会话态与后续代理行为

## 非目标

- 不在此阶段追求完整通用自治代理
- 不在此阶段覆盖所有桌面自动化场景
- 不在此阶段做重度多模态模型推理

## 设计原则

- 浏览器操控必须有会话态，而不是一次性工具调用堆叠
- 视觉输入应先以 OCR 文本为主，不立即扩展到复杂图像理解
- 插件 / MCP 管理必须强调诊断能力
- 关键权限、超时、目录位置应配置化

## 浏览器能力演进

当前浏览器相关能力主要偏：

- 打开
- 搜索
- 简单自动化

Phase 6 应增强为：

- 持续页面上下文
- 多步页面交互
- 更自由的导航、点击、提取、等待
- 浏览器状态可进入会话态

## 视觉能力范围

第一阶段视觉建议只做：

- OCR 获取屏幕文本
- 结构化提取当前屏幕可见关键信息
- 将视觉摘要送入 `SessionContext` 或本轮运行态

不建议第一版就直接追求复杂图像语义。

## 插件 / MCP 管理范围

建议增强：

- 插件启停状态可视化
- MCP 连接状态与工具数量
- 错误诊断信息
- 权限与能力边界展示

## 配置建议

以下关键参数应配置化：

- 浏览器自动化默认超时
- 页面等待超时
- OCR 扫描频率
- OCR 截图区域 / 模式
- MCP 连接重试参数
- 浏览器截图 / OCR 中间产物目录

## 与前置阶段的依赖

Phase 6 高度依赖：

- Phase 4 的会话态与线程模型
- Phase 5 的交互模式与输入输出稳定性

因为浏览器和视觉一旦接入，会显著增加：

- 运行时状态量
- 事件数量
- 上下文预算压力

## 验收标准

- 插件与 MCP 管理界面具备更清晰的启停与诊断能力
- 浏览器插件可支持连续操作场景
- OCR 可稳定获取屏幕文本
- 视觉结果可进入后续对话或代理任务
