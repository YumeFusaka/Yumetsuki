# Phase 8 / Phase 9 阶段设计草案

> 日期：2026-05-28
>
> 状态：设计草案。本文用于保留第二轮头脑风暴后的阶段意图，不是可执行级实施计划。

## 目的

Phase 7 已经收敛为“可诊断性与最小恢复闭环”。Phase 8 和 Phase 9 的能力都依赖 Phase 7 中的日志 trace、诊断包、配置健康、工具审计和聊天窗失败恢复，因此现在不适合提前写完整 implementation plan。

本文只做两件事：

- 把 Phase 8 的目标、边界、依赖、验收方式先设计清楚，避免 Phase 7 完成后丢失上下文。
- 把 Phase 9 保持为路线图级草案，只定义入口条件、主题和风险，不承诺具体实现顺序。

## 总体阶段关系

```text
Phase 7：可诊断性与最小恢复闭环
→ Phase 8：记忆可信度、交互闭环与低打扰陪伴
→ Phase 9：生态、Computer Use、视觉与演出扩展
```

关键原则：

- Phase 8 不绕过 Phase 7 的诊断底座。记忆、会话画像、语音状态和主动行为都必须能通过日志追踪来源与失败原因。
- Phase 9 不绕过 Phase 7 的工具审计。插件、MCP、浏览器、OCR 和 Computer Use / 桌面应用自动化继续坚持显式触发、权限确认和可追踪。
- 不把实验路径改成默认稳定路径。`wav + inline` 继续作为 TTS 保底组合；PCM、会话参考、情绪参数和 Computer Use / 桌面应用自动化都必须显式启用。
- 不做后台持续看屏，不做无人值守高权限操作，不把桌宠改造成不受控的远程控制器。

## Phase 7 依赖门槛

Phase 8 正式拆实施计划前，至少需要 Phase 7 提供以下稳定接口或行为：

- 平台日志事件具备 `trace_id`、`request_id`、`session_id` 和 `stage`，并支持查询与导出。
- 诊断包能脱敏导出，不包含 API key、私有 URL / 私有服务地址全文、URL 凭据、截图原图、音频、本地模型完整路径或长 OCR 原文；URL 只能保留脱敏摘要。
- 工具调用审计至少记录来源、工具名、权限、参数摘要、结果摘要、耗时、失败原因和关联 trace。
- 配置健康检查能区分 error / warn / info，并且不自动修复用户配置。
- 诊断运行器和诊断页入口已落地；真实服务与设备检查保持手动触发，且可 mock。
- API、TTS、STT、OCR、MCP、Playwright Edge 的手工矩阵结果可记录、可关联平台日志。
- 聊天窗失败后保留输入、可重试、可打开平台日志，并能把失败事件关联到 trace。

如果这些门槛未满足，Phase 8 只能继续停留在设计层，不进入正式实施计划。

## Phase 8：记忆可信度、交互闭环与低打扰陪伴

### 阶段目标

Phase 8 的目标不是让桌宠“功能更多”，而是让长期陪伴更可信、每一轮交互更可恢复、主动行为更低打扰。

用户应该能感受到：

- 桌宠记住的内容有来源，不会轻易把临时话题当长期偏好。
- 回复失败、语音失败、TTS 播放失败时，有清楚的恢复路径。
- 主动发言不是随机打扰，而是基于上下文、冷却、状态和触发理由。

### 工作流 A：记忆质量治理与可回溯记忆账本

目的：

- 在写入长期记忆前引入“记忆候选”概念。
- 为每条候选记录来源、类型、置信度、升格原因、冲突状态和跳过原因。
- 对话日志或平台日志能解释“为什么记了 / 为什么没记”。

第一版建议范围：

- 候选来源：当前 turn、近期 `SessionContext`、显式“记住”表达、反思结果。
- 候选类型：用户偏好、事实信息、关系状态、长期约束、短期任务。
- 升格策略：显式记忆优先；重复内容跳过；明显短期约束不升格；低置信度只记录候选不写入长期记忆。
- 冲突策略：第一版只标记冲突并记录，不自动删除旧记忆。

不做：

- 不做复杂人格成长数值。
- 不做自动批量删除长期记忆。
- 不把模型自由生成的低置信内容直接写入 Mem0。

依赖：

- Phase 7 日志 trace 与诊断包。
- `SessionPolicy.collect_mem0_candidates`
- `Mem0MemoryStore.add_memory`
- `LogService`
- 对话日志页的记忆标签展示能力。

验收方向：

- 重复候选会跳过并记录原因。
- 短期约束不会升格为长期记忆。
- 显式“记住”比普通推断有更高优先级。
- 记忆写入、跳过、冲突和失败都能通过 trace 回看。

### 工作流 B：会话状态画像与关系连续性

目的：

- 让 `SessionSummary.mood_state`、`relationship_state`、`current_topic` 参与回复，但不增加首字前 LLM 调用。
- 回复后异步更新会话画像，下一轮热路径只读取已有摘要。

第一版建议范围：

- 用户情绪：只记录高置信、近期有效的轻量标签。
- 关系状态：只记录与角色互动风格相关的短摘要，不做数值亲密度。
- 当前主题：用于下一轮上下文连续，不进入长期记忆，除非被记忆治理明确升格。
- 更新时机：回复完成后异步更新；失败或取消时不写入新画像。

不做：

- 不在每轮首字前额外调用 LLM 做画像分析。
- 不把用户情绪判断写成长期事实。
- 不做强医疗、心理诊断式判断。

依赖：

- Phase 7 trace 日志。
- `SessionContext.summary`
- `AgentReflector`
- `SessionPolicy`
- 主动行为 prompt 构造链路。

验收方向：

- 画像更新不阻塞首字响应。
- 低置信结果被丢弃或只保留短期。
- prompt 注入可以被测试验证。
- 状态变化有日志事件，能解释来源和置信度。

### 工作流 C：聊天窗完整微交互闭环

目的：

- 在 Phase 7 最小失败恢复基础上，补齐常见恢复动作。
- 让用户不用进入设置页也能处理最近一轮回复。

第一版建议范围：

- 最近一轮操作区：重试、停止、复制角色回复、打开平台日志。
- TTS 操作：最后一句重播、停止说话、失败后不阻塞文本阅读。
- 被动气泡：点击后展开最近上下文，返回主对话状态。
- 输入恢复：失败后保留用户输入，重试时沿用同一 trace 或派生 request。

不做：

- 不把聊天窗做成工具控制台。
- 不把所有操作都堆到主界面；优先使用状态条、右键菜单或最近一轮操作区。
- 不做多轮复杂编辑历史。

依赖：

- Phase 7 聊天失败恢复。
- `ChatWindow`
- TTS pipeline
- 对话日志与平台日志入口。

验收方向：

- 忙碌态停止、失败重试、打开日志、复制回复都有 Qt offscreen 测试。
- TTS 重播不重新触发 LLM。
- 被动气泡点击恢复主对话，不破坏被动状态规则。
- 长文本滚动、缩放和状态条不互相遮挡。

### 工作流 D：低延迟语音对话状态机

目的：

- 优化“录音 -> STT -> Agent -> TTS”的状态反馈、取消语义和耗时指标。
- 近期只做状态机，不做真正流式 STT。

第一版建议范围：

- 状态：录音中、识别中、思考中、回复中、语音合成中、说话中、停止中、失败可重试。
- 指标：录音结束到 STT 完成、STT 完成到首字、首字到首音频。
- 取消：用户停止后迟到结果必须被忽略，TTS 队列和临时音频要清理。
- 默认路径：继续保留 `wav + inline` 作为稳定保底。

不做：

- 不引入真正边说边识别。
- 不把 PCM 流式作为默认稳定路径。
- 不让语音链路绕过普通聊天的 Agent、SessionContext、日志和 TTS 管线。

依赖：

- Phase 7 trace 与聊天失败恢复。
- `STTRecorder`
- `STTManager`
- `TTSPipelineController`
- `ChatWindow` 状态条。

验收方向：

- 状态机测试覆盖成功、失败、超时、取消和迟到结果。
- 耗时指标进入平台日志。
- 真实设备测试只作为手工矩阵，不作为 pytest 硬依赖。

### 工作流 E：日常陪伴循环 / 主动行为 2.0

目的：

- 把主动行为从“闲置后说一句话”升级为低打扰的日常陪伴闭环。
- 主动行为必须能解释触发理由，并遵守冷却、活跃时段和被动状态约束。

第一版建议范围：

- 触发来源：空闲时长、最近主题、会话画像、用户互动节奏、低风险轻量提醒。
- 触发类型：问候、陪伴、轻量休息提醒、承接最近话题。
- 触发理由：写入平台日志，用户能从日志看到为什么主动开口。
- 冷却策略：全局冷却 + 同类事件冷却 + 用户活跃时抑制。

不做：

- 不做完整日程软件。
- 不做强制提醒或连续打扰。
- 不默认调用高权限工具。

依赖：

- 记忆账本。
- 会话状态画像。
- Phase 7 trace 日志。
- `ProactiveScheduler`
- 被动气泡与 TTS 链路。

验收方向：

- mock 时间和上下文验证触发、抑制、冷却和日志。
- 主动消息仍复用普通聊天的切句、TTS 和情绪标签链路。
- 用户忙碌或近期拒绝主动发言时，主动行为降频。

## Phase 8 推荐执行顺序

1. 记忆质量治理与可回溯记忆账本。
2. 会话状态画像与关系连续性。
3. 聊天窗完整微交互闭环。
4. 低延迟语音对话状态机。
5. 日常陪伴循环 / 主动行为 2.0。

理由：

- 记忆可信度和会话画像是主动行为质量的前置条件。
- 聊天窗微交互可以并行设计，但实现时应依赖 Phase 7 的失败恢复接口。
- 语音状态机需要 TTS / STT 的诊断指标，适合在 trace 稳定后推进。
- 主动行为 2.0 最容易放大错误记忆和错误画像，因此放在 Phase 8 后段。

## Phase 8 升级为实施计划的条件

满足以下条件后，可以把本草案拆成一个或多个实施计划：

- Phase 7 已实现并通过聚焦自动化回归。
- 日志事件字段、工具审计字段和诊断包格式已经稳定。
- 诊断运行器和诊断页入口已落地，API / TTS / STT / OCR / MCP / Playwright Edge 的手工矩阵可记录、可关联日志。
- 聊天窗最小失败恢复已落地。
- 对 Phase 8 的第一批目标达成一致，优先建议先拆“记忆质量治理”单独计划。

## Phase 9：生态、Computer Use、视觉与演出扩展

### 阶段定位

Phase 9 是能力边界扩展阶段，不应抢在诊断、审计、记忆可信度和交互闭环之前。

它的核心问题是：在已有能力可追踪、可确认、可恢复之后，如何让插件、MCP、浏览器、桌面应用、视觉和角色演出变得更强，但不失控。

其中 **Computer Use / 桌面应用自动化** 是 Phase 9 的核心主题之一。它不是简单增加几个工具，而是让 Yumetsuki 能在用户确认和审计保护下，观察当前电脑状态，并通过系统动作、应用接口、浏览器自动化、Windows UI Automation、OCR 和必要的鼠标键盘动作去使用应用。

### 入口条件

Phase 9 进入实施前，至少应满足：

- Phase 7 诊断与工具审计已落地。
- Phase 8 的记忆账本和会话画像至少完成第一版。
- 浏览器、OCR、MCP、STT / TTS / API 的手工实机矩阵已有记录。
- 高权限工具调用有明确用户确认、取消和失败恢复路径。
- 用户能随时停止正在执行的桌面控制任务，停止后迟到结果不会继续操作电脑。

### Computer Use 能力分层

Phase 9 不应直接承诺“任意应用都能稳定控制”。桌面应用自动化按能力层逐步推进：

| 层级 | 能力 | 稳定性 | Phase 9 定位 |
|---|---|---|---|
| L1 系统动作 | 打开应用、切换窗口、快捷键、剪贴板、音量、媒体键 | 高 | 可作为基础能力 |
| L2 应用专用接口 | URL scheme、CLI、官方 API、已知应用协议 | 高到中 | 优先使用 |
| L3 Windows UI Automation | 读取窗口控件、按钮、输入框、菜单并操作 | 中 | MVP 核心 |
| L4 视觉 + OCR + 坐标点击 | 看屏幕、识别文字 / 按钮、鼠标点击 | 中到低 | 只作为兜底和显式确认路径 |
| L5 自主任务代理 | 观察 -> 规划 -> 执行 -> 校验 -> 失败恢复 | 难度高 | Post-1.0 或 Phase 9 后段探索 |

第一版 Computer Use MVP 应优先覆盖 L1、L2、L3；L4 只做受限兜底；L5 不进入 1.0 验收门。

能力层级不是权限等级。每个 Computer Use 动作都必须同时记录能力层级、风险等级、插件来源、最低权限和是否需要用户确认。

| 动作类型 | 能力层级示例 | 最低权限建议 | 确认要求 |
|---|---|---|---|
| 只读窗口枚举、当前窗口标题、自动化会话状态 | L1 / L3 | low | 通常不需要，但要审计 |
| 打开应用、窗口聚焦、媒体键、低风险快捷键 | L1 | low / medium | 可按用户设置决定 |
| URL scheme、CLI 或官方 API 的只读查询 | L2 | medium | 通常不需要，但要审计 |
| UI Automation 填写、点击、选择控件 | L3 | high | 需要逐步确认 |
| 截图 / OCR / 坐标点击兜底 | L4 | high | 必须显式区域、显式确认 |
| 多步自主任务代理 | L5 | high | 不进入 1.0 验收门 |

运行时隐私规则：

- 不主动读取密码框、2FA、token、密钥、支付信息或私聊内容。
- 剪贴板只允许用户显式触发的单次读取 / 写入，不做后台轮询。
- 截图和 OCR 必须区域限定、文本截断、敏感模式脱敏，并审计来源区域。
- 视觉观察结果进入模型上下文前必须经过摘要和脱敏，不保留截图原图。

### 路线主题 A：插件 SDK 开发者体验增强

目标：

- 让个人插件更容易创建、测试、诊断和维护。
- 继续兼容现有 `example_echo`、`system_control`、`web_automation` 插件。

可能范围：

- 插件模板。
- manifest 校验与错误提示。
- 参数 schema 诊断。
- 权限声明说明。
- 插件测试辅助工具。

后置原因：

- 需要 Phase 7 工具审计字段稳定后，再确定插件 SDK 应暴露哪些调试信息。

### 路线主题 B：MCP 兼容性实机模板

目标：

- 为 stdio、SSE、HTTP 等 MCP transport 建立可复制的实机模板和错误分类。

可能范围：

- 本地 mock MCP server。
- 常见失败分类：启动失败、连接超时、请求超时、协议错误、工具返回异常。
- MCP 设置页的诊断提示。

后置原因：

- 需要 Phase 7 配置健康和诊断 runner 先落地，避免模板变成另一套独立检查系统。

### 路线主题 C：显式读屏区域选择与视觉摘要

目标：

- 在坚持显式触发的前提下，提高 OCR 使用效率。
- 支持用户选择屏幕区域，减少无关文本进入上下文。

可能范围：

- 区域选择 UI。
- OCR 文本摘要。
- `SessionContext.visual_observations` 的来源区域信息。
- 视觉输入日志脱敏与截断。

不做：

- 不做后台持续看屏。
- 不保存截图原图到诊断包。
- 不引入复杂图像语义推理作为默认能力。

后置原因：

- 需要 Phase 7 诊断包脱敏规则和 Phase 8 会话状态使用方式先稳定。

### 路线主题 D：情绪驱动演出调度器

目标：

- 把 `[emotion:...]` 从立绘切换升级为统一演出信号。
- 控制立绘、气泡停留、TTS 节奏和主动行为语气，但保持所有扩展可回退。

可能范围：

- 情绪事件模型。
- 角色资源缺失回退。
- TTS 风格参数显式扩展。
- 主动行为和被动气泡的演出策略。

不做：

- 不要求所有 TTS 服务支持情绪参数。
- 不破坏原版 GPT-SoVITS 路径。
- 不让情绪状态覆盖用户明确指令。

后置原因：

- 依赖 Phase 8 会话画像、主动行为 2.0 和 TTS 稳定状态机。

### 路线主题 E：Computer Use / 桌面应用自动化

目标：

- 在工具审计和权限确认基础上，让 Yumetsuki 能真正使用电脑上的应用，但每一步都可观察、可确认、可审计、可停止。

可能范围：

- 窗口枚举、窗口聚焦、窗口标题和进程信息读取。
- 系统动作：快捷键、剪贴板辅助、媒体键、音量、打开应用。
- 应用专用接口：优先使用 URL scheme、CLI、官方 API 或稳定本地协议。
- Windows UI Automation：读取按钮、菜单、输入框、列表项，执行点击、填写、选择等控件级动作。
- 视觉兜底：截图、OCR、区域识别和坐标点击，但必须显式确认，且默认只用于没有稳定控件接口的应用。
- 自动化步骤计划展示：执行前展示将要操作的应用、窗口、动作和风险等级。
- 只读观察步骤可自动执行；低风险写入、表单草稿、窗口切换、复制粘贴等动作可在用户确认后执行。
- 硬禁止事项必须由用户亲自完成：读取或输入密码、支付 / 转账 / 下单、发送消息 / 邮件 / 发布内容、不可恢复删除、安装软件、执行系统命令。
- 失败中止、用户取消、迟到结果忽略和恢复记录。

不做：

- 不承诺任意应用 100% 稳定可控。
- 不做无人值守高权限桌面自动化。
- 不通过 Computer Use 自动输入密码、支付、删除文件、发消息或执行系统命令；如确需命令执行，只能走现有 `system_control.run_command` 的 high 权限和专门确认链路。
- 不自动重放高权限工具或高风险桌面动作。
- 不把网页、MCP、系统命令、UI Automation 和坐标点击混成不可审计的黑盒流程。

停止验收标准：

- 用户取消后，未开始步骤全部丢弃。
- 进行中动作只允许完成最小安全收尾，不能继续点击、输入或触发下一步。
- 迟到结果只能写日志，不能触发后续动作。
- 自动化浏览器、UI Automation、OCR 和鼠标键盘相关资源必须释放。

后置原因：

- 必须等待 Phase 7 工具审计、诊断包、配置健康和 Phase 8 交互恢复路径成熟。
- 真正控制电脑会放大权限风险、误操作风险和界面不稳定风险，必须建立在可停止、可追踪、可恢复的基础上。

### 路线主题 F：发布打包与兼容性矩阵

目标：

- 在核心链路通过实机矩阵后，再整理发布包、依赖检查和兼容性说明。

可能范围：

- Playwright Edge、RapidOCR、faster-whisper、音频设备、TTS 服务兼容矩阵。
- 发布前诊断报告模板。
