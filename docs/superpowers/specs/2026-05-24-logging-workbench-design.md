# 日志工作台设计

日期：2026-05-24

> 状态：设计已确认，可进入实施计划阶段。

## 背景

当前 Yumetsuki 已经具备：

- Agent 事件发布
- `AgentPage` 里的运行日志 Tab
- TTS / LLM / Tool 等多条内部链路

但现有日志体系仍然存在明显问题：

- 日志入口放在 `Agent` 设置页里，职责混乱
- 现有“运行日志”只覆盖部分 Agent 事件
- TTS 切分、句段入队、发送 / 收回、播放状态等关键排障信息不可见
- 日志当前主要是 UI 字符串输出，缺少统一结构化模型
- 缺少默认持久化，重启后难以回溯问题
- 时间信息不统一，不利于分析时序问题

这会直接影响 Phase 4-6 的开发效率，尤其在排查：

- TTS 重复播报
- 文本切分异常
- 工具调用时序问题
- 服务端请求 / 响应边界问题
- UI 卡顿与事件积压

因此需要把当前零散日志升级成一套明确分层、可持久化、可回看的日志工作台。

## 目标

- 把日志从 `Agent` 设置中抽离，成为独立工作台
- 明确区分“面向阅读的对话日志”和“面向排障的系统日志”
- 为关键内部链路提供统一结构化日志事件
- 系统日志默认持久化，便于开发中回溯问题
- 所有日志项统一带时间戳
- 默认尽量全量记录，但自动脱敏敏感信息

## 非目标

- 不在首版引入数据库型日志系统
- 不在首版支持远程日志上传
- 不在首版做全文索引或复杂分析面板
- 不把日志系统扩展成通用监控平台

## 核心设计结论

推荐采用“两层日志体系 + 独立导航页”的方案：

- 左侧导航新增：
  - `对话日志`
  - `系统日志`
- `Agent` 页面移除现有“运行日志”Tab，仅保留 Agent 配置能力
- 底层增加统一结构化 `log_service`
- `对话日志` 与 `系统日志` 共用同一套事件模型，但呈现和持久化策略不同

这样可以同时满足：

- 普通使用时可阅读
- 开发排障时可深入
- 后续 STT / 浏览器 / 视觉能力继续接入时无需重做日志基础设施

## 日志分层

## 1. 对话日志

定位：面向阅读和回看，强调“会话时间线”。

主要展示：

- 用户输入
- 角色回复
- Thinking 预览
- 工具调用摘要
- 关键状态提示
  - 例如：切换到多步推理、TTS 跳过若干句段、工具执行失败等

原则：

- 保持轻量
- 连续可读
- 不直接暴露大量底层技术噪声
- 默认按会话组织

## 2. 系统日志

定位：面向开发排障，强调“内部链路和时序可见”。

建议覆盖：

- 文本切分结果
- TTS 句段入队 / 跳过 / 取消 / 超时 / 播放
- TTS 请求发送与响应摘要
- LLM 流式开始 / 结束 / Thinking / 工具调用链路
- Tool / Plugin / MCP 调用开始、结束、错误、耗时
- Agent 路由、反思、多步推理内部事件
- UI bridge、队列积压、关键异常堆栈

原则：

- 默认持久化
- 尽量全量
- 默认自动脱敏
- 不记录原始音频二进制

## 导航与页面结构

设置中心左侧导航调整为：

1. `API 设定`
2. `角色管理`
3. `记忆`
4. `对话日志`
5. `系统日志`
6. `Agent`
7. `插件`
8. `系统`

说明：

- `对话日志` 和 `系统日志` 都是运行期工作台，不属于配置页
- `Agent` 页面只保留：
  - 规划
  - 反思
  - 多步推理
  - 主动行为
  - 以及后续真正属于 Agent 行为配置的内容

## 结构化日志模型

推荐统一底层事件模型 `LogEvent`：

- `id`
  - 唯一标识
- `timestamp`
  - 精确到毫秒
- `channel`
  - `conversation`
  - `system`
- `level`
  - `debug`
  - `info`
  - `warn`
  - `error`
- `source`
  - 例如：
    - `chat.tts`
    - `chat.segmenter`
    - `llm.manager`
    - `tool.registry`
    - `agent.planner`
- `event_type`
  - 例如：
    - `tts.segment_enqueued`
    - `tts.segment_skipped`
    - `tts.request_sent`
    - `tts.response_received`
    - `llm.stream_started`
    - `tool.call_started`
- `session_id`
  - 用于串起单次会话
- `utterance_id`
  - 可选，用于定位单轮回复
- `summary`
  - UI 列表直接展示的短文本
- `details`
  - 结构化上下文
- `sensitive`
  - 标记是否包含已脱敏字段

## 持久化设计

## 1. 系统日志

默认持久化到本地文件，按日期切换：

- `data/logs/system/YYYY-MM-DD.jsonl`

格式：

- 一行一个 JSON
- 便于追加写入
- 便于外部工具筛选与定位问题

## 2. 对话日志

推荐按会话持久化：

- `data/logs/conversation/<session_id>.jsonl`

说明：

- 与系统日志共享事件模型
- 但只保留 `conversation` 渠道事件
- 便于用户按单次聊天回看

## 3. 首版文件策略

- 系统日志：按天切文件
- 对话日志：按 `session_id` 切文件
- 不在首版实现复杂归档策略
- 允许后续增加保留天数和自动清理

## 脱敏规则

系统日志要求“尽量全量，但不泄露敏感信息”。

默认脱敏范围：

- `api_key`
- `token`
- `password`
- `authorization`
- `cookie`
- 敏感配置中的真实密钥和凭证

默认不直接落盘：

- 原始音频二进制
- 完整敏感请求体
- 完整 headers 中的认证信息

音频相关建议只记录：

- 字节大小
- 预计时长
- sample rate
- channels
- 模式（`wav` / `pcm_stream`）

文本相关建议：

- 对话日志保留可读正文
- 系统日志默认记录摘要、长度、关键字段
- 若未来需要完整正文采样，应通过显式“开发者详细模式”开启，而不是默认开启

## UI 设计

## 1. 对话日志页

目标：像时间线，而不是调试终端。

建议结构：

- 顶部控制栏：
  - 当前会话选择
  - 搜索
  - 导出当前会话
  - 清空当前视图
- 主体：
  - 时间线列表
  - 每条都带完整时间戳
- 每条内容：
  - `[2026-05-24 21:14:03.182] [用户] ...`
  - `[2026-05-24 21:14:05.021] [角色] ...`
  - `[2026-05-24 21:14:05.118] [Thinking] ...`
  - `[2026-05-24 21:14:05.602] [工具摘要] ...`

## 2. 系统日志页

目标：像轻量排障台，而不是配置页附属文本框。

建议结构：

- 顶部控制栏：
  - 日志启用开关
  - 等级筛选
  - 来源筛选
  - 会话筛选
  - 搜索框
  - 打开日志目录
  - 导出当前筛选结果
  - 清空当前视图
- 主列表：
  - 每条都带完整时间戳
  - 例如：
    - `[2026-05-24 21:14:03.182] [INFO] [chat.tts] segment_enqueued | utterance=3 segment=5 text="..." `
- 详情能力：
  - 复制摘要
  - 复制完整 JSON
  - 复制脱敏详情

## 接线点

首版优先接入以下链路：

## 1. `ui/chat/window.py`

- 文本切分
- `_enqueue_tts_segment`
- `segment skipped / cancelled / timed_out / failed / played`
- 新轮开始时旧轮压制
- WAV / PCM 播放启动与完成

## 2. `tts/adapters/gptsovits.py`

- 请求发送前摘要
- 请求模式
- 响应返回摘要
- 超时 / 错误 / 回退

## 3. `llm/manager.py`

- 流式开始 / 完成
- thinking 事件
- 内容累计长度
- tool call 进入 / 退出

## 4. `agent/manager.py`

- 复用现有 Agent 事件
- 增加对话日志友好的摘要事件

## 5. `core/tool_registry.py` 及插件调用链

- 调用开始
- 参数摘要
- 结果摘要
- 错误
- 耗时

## 6. `core/event_bus.py`

- 不作为日志核心存储点
- 只继续承担事件分发角色

## 首版功能范围

首版建议直接支持：

- `对话日志` 导航页
- `系统日志` 导航页
- 统一结构化 `log_service`
- 系统日志持久化
- 对话日志按会话组织
- 统一时间戳
- 自动脱敏
- 导出
- 按日期切系统日志文件
- 一键复制摘要 / 详情

首版明确不做：

- 日志数据库
- 远程上传
- 复杂查询语言
- 全文索引
- 可视化统计面板

## 推荐实现顺序

1. 建立统一 `log_service`
2. 建立 `LogEvent` 结构与脱敏器
3. 完成 JSONL 持久化
4. 接 TTS / LLM / Tool 三条关键链路
5. 新增 `对话日志` 页与 `系统日志` 页
6. 从 `Agent` 页面移除“运行日志”
7. 增加导出、复制详情、打开目录等增强能力

## 风险

### 风险 1：继续沿用字符串日志

如果仍以“页面自己拼字符串”的方式扩展：

- 难以持久化
- 难以筛选
- 难以脱敏
- 后续字段演进成本高

### 风险 2：日志页继续挂在 Agent 下

如果只把现有 `Agent` 页继续做大：

- 配置页职责继续混乱
- 后续日志页可维护性会更差

### 风险 3：全量记录但没有脱敏

如果系统日志追求全量而无默认脱敏：

- 极易误写入真实敏感信息
- 后续分享日志排障会有风险

## 测试要求

- 日志事件脱敏测试
- JSONL 追加写入测试
- 系统日志按日期切文件测试
- 对话日志按会话归档测试
- 日志 UI bridge 与批量刷新测试
- `对话日志` 过滤正确性测试
- TTS / Tool / LLM 接线后关键事件存在性测试

## 验收标准

- 设置中心导航中出现独立 `对话日志` 与 `系统日志`
- `Agent` 页面不再承载运行日志
- 系统日志默认持久化，重启后可回看
- 所有日志项附带时间戳
- 系统日志可看到：
  - 文本切分
  - TTS 发送 / 收回
  - 工具调用
  - 关键内部事件
- 默认不会泄露真实敏感信息
- 支持导出、复制详情、打开日志目录
