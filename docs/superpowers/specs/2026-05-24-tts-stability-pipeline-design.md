# TTS 稳定性与有界流水线设计

日期：2026-05-24

> 状态：设计已确认，可在 Phase 4 中实施。

## 背景

当前 Yumetsuki 已实现：

- 句级增量 TTS
- 输出语言约束
- 逐句翻译
- PCM / WAV 模式

但在长时间使用中仍存在高风险问题：

- PCM 流式请求可能长时间挂起
- 某个句段卡住会堵住后续句段
- 分段翻译 worker 和 TTS worker 没有明确并发上限
- 新一轮输入到来时，旧轮残余任务可能继续占用资源

这些问题会直接表现为：

- “传回 TTS 时无响应卡死”
- “越用越卡”
- “前面一句出问题，后面都不播”

## 目标

- 把 TTS 从“可用但脆”提升为“长时间运行可恢复”
- 为每个句段提供明确生命周期
- 给翻译与合成链路增加并发上限和取消语义
- 让失败句段可以推进，不再无限阻塞后续句段

## 非目标

- 不在此阶段新增新 TTS 引擎
- 不在此阶段实现 STT 双工联动
- 不在此阶段追求最优质音频，只优先稳定性和恢复能力

## 设计原则

- 不允许无限等待
- 不允许无上限 worker 扩散
- 句段播放顺序必须稳定
- 前序句段失败时，后序句段必须能继续
- 所有关键阈值应可配置，不要写死

## 句段生命周期

每个句段建议具有明确状态：

- `queued`
- `translating`
- `ready_for_tts`
- `synthesizing`
- `streaming`
- `played`
- `failed`
- `timed_out`
- `skipped`
- `cancelled`

作用：

- 便于调试
- 便于推进播放状态机
- 便于后续接入 STT 打断与被动互动

## 有界流水线

建议把 TTS 链路拆成固定流水线：

1. 句段切分
2. 翻译队列
3. 合成队列
4. 播放队列

### 推荐约束

- 翻译 worker 有上限
- 合成 worker 有上限
- 播放始终单通道顺序推进

### 为什么这样做

当前的问题不是“不异步”，而是“异步失控”。  
正确做法不是继续增加线程，而是增加背压。

## 超时策略

### 1. 连接超时

保持连接建立阶段的超时控制。

### 2. 读超时

PCM 流式场景必须有有限读超时。  
不允许 `None` 式无限等待。

### 3. 单句段总超时

即使连接不断开，如果某句段总耗时过长，也必须进入 `timed_out`。

## 失败推进策略

当句段失败时：

- 当前句段进入 `failed` / `timed_out`
- 播放游标推进到下一句段
- 后续句段继续尝试播放

不能因为前一段失败，让整轮全堵住。

## 取消语义

当用户发起新一轮输入时：

- 旧轮未完成句段应尽快进入 `cancelled`
- 旧轮翻译 / 合成 / 播放链路不应继续占用主状态机
- 旧结果即使返回，也只允许被丢弃，不再影响新轮

## 配置建议

以下关键数值应配置化：

- 翻译并发上限
- 合成并发上限
- PCM 读超时
- 单句段总超时
- 队列最大长度
- 失败后是否强制某会话回退到 WAV

建议后续配置入口：

- `data/config/api.yaml` 中的 TTS 项
- 或 `agent.yaml` / 专门音频运行配置

## 与 SessionContext 的关系

TTS 本身不应进入首字热路径。  
但 TTS 句段状态应具备可回流的任务状态能力，以便：

- 调试观察
- 后续和 STT 联动
- 必要时把关键失败结果写入会话态

## 验收标准

- 长时间使用不再出现明显无响应卡死
- 单句段失败后后续句段仍可播放
- 新轮输入能明确压制旧轮残留播报
- 翻译 / 合成 worker 数量不再无上限增长
