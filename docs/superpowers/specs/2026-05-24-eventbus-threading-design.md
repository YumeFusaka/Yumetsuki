# EventBus 与线程治理设计

日期：2026-05-24

> 状态：设计已确认，可在 Phase 4 中实施。

## 背景

当前 Yumetsuki 的事件流已经覆盖：

- Agent 路由
- 记忆检索
- 工具执行
- LLM 开始 / 完成
- 反思完成
- 多步推理进度

但 `EventBus` 仍然是同步直调模型，发布线程与订阅线程没有明确隔离。随着后续的：

- `SessionContext`
- TTS 有界流水线
- STT
- 被动互动
- 视觉输入

持续接入，当前模型会越来越容易出现：

- 跨线程 UI 调用
- 日志监听拖慢主链路
- 事件顺序不稳定
- 某一类事件过载后拖累全局

因此需要在 Phase 4 中，把事件系统从“能用”提升到“线程边界明确、扩展成本可控”。

## 目标

- 明确 UI 事件、状态事件、日志事件的职责边界
- 让后台线程不直接驱动 UI 更新
- 保持当前功能兼容，不重写整套 UI
- 为后续 STT、视觉、浏览器自由操控预留稳定事件基础

## 非目标

- 不在此阶段引入完整外部消息队列
- 不在此阶段做分布式事件系统
- 不要求所有事件都持久化

## 现状问题

当前模型的主要问题：

- `EventBus.publish()` 同步遍历订阅者
- 订阅者是否触发 UI 更新取决于调用点和实现细节
- 日志事件与控制事件未分层
- 没有统一的主线程分发入口

这意味着：

- 事件正确性依赖调用方自觉
- 某些页面监听逻辑会和后台线程时序产生隐式耦合
- 后续功能越多，越容易出现“看起来没崩，但体感越来越卡”的问题

## 设计原则

- 后台线程只发布事件，不直接操作 UI
- UI 相关事件统一回到主线程消费
- 日志事件允许弱一致，但控制事件必须保证顺序
- 事件通道应可分层，而不是所有事件共享同一优先级
- 关键队列长度、合并策略、刷新节流应做成可配置项

## 目标模型

推荐拆成三类事件：

### 1. 控制事件

用于驱动真实状态变化：

- 当前任务状态变化
- 工具执行完成
- 会话上下文更新完成
- TTS 句段状态变化

要求：

- 顺序清晰
- 不允许随意丢弃
- 可追踪来源

### 2. UI 事件

用于界面呈现：

- 聊天窗口状态提示
- 设置页监控展示
- 被动互动显示

要求：

- 必须在主线程消费
- 允许节流
- 必要时可合并重复刷新

### 3. 日志 / 观测事件

用于调试与诊断：

- planner 判定记录
- 记忆检索记录
- 工具结果摘要
- TTS 异常日志

要求：

- 不允许拖慢主链路
- 允许弱一致
- 可以做批量刷新或限速

## 推荐实现方向

### 方案 A：保留 EventBus 接口，增加线程安全与快照发布

第一步只修基础安全性：

- 订阅 / 退订加锁
- 发布时复制 handler 快照
- 避免遍历中修改订阅列表

作用：

- 先解决最直接的不安全问题

### 方案 B：增加主线程分发桥

增加一个 Qt 主线程桥接器，例如：

- `UIEventBridge`
- 或统一 `QObject + Signal`

职责：

- 后台线程发布的 UI 事件先进入桥
- 桥在主线程中转发给真正界面订阅者

作用：

- 明确 UI 更新线程归属

### 方案 C：日志事件独立节流

对日志类事件增加：

- 合并策略
- 队列长度限制
- 刷新频率限制

可配置项建议：

- `event_bus.log_max_buffer`
- `event_bus.log_flush_interval_ms`
- `event_bus.log_drop_policy`

作用：

- 防止调试日志反过来拖慢功能链路

## 配置建议

以下关键数值不要写死，建议后续接入配置：

- 主线程 UI 刷新节流间隔
- 日志缓冲区长度
- 日志刷新间隔
- 每类事件队列长度上限
- 是否启用某类调试事件

这些值可进入：

- `data/config/agent.yaml`
- 或专门的运行时调试配置

## 对 Phase 4 的价值

完成后，Phase 4 其他模块会直接受益：

- `SessionContext` 更新不再通过隐式跨线程副作用通知 UI
- TTS 状态机和播放事件可稳定回流主线程
- 设置页日志监控不再绑定后台线程执行路径

## 风险

### 风险 1：只加锁，不分层

如果只做线程安全，不做事件分类，后面仍然会出现：

- UI 过刷
- 日志噪声拖慢界面
- 控制事件和展示事件混在一起

### 风险 2：事件过度设计

如果一开始就把事件系统做成复杂总线，会让 Phase 4 失焦。

建议顺序：

1. 先线程安全
2. 再主线程桥
3. 再日志节流

## 验收标准

- 后台线程不再直接推动 UI 更新
- 设置页和聊天界面的监听路径线程边界明确
- 日志高频事件不会显著拖慢主链路
- Phase 5 / 6 可在此基础上继续扩展事件种类
