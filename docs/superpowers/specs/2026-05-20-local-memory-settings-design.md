# 本地记忆与设置窗口复用设计

日期：2026-05-20

## 目标

解决当前桌宠应用中的两个具体问题：

1. 修复聊天窗口右键菜单中的“打开设置”会无限创建新设置窗口的问题。
2. 将当前偏云端的记忆初始化方式替换为可配置的本地记忆方案，基于 Mem0 OSS + 本地 Chroma 持久化，同时保留主对话 LLM 继续使用现有远程提供方。

## 范围

本设计涵盖：

- 设置窗口复用与前台激活行为
- 本地记忆配置模型、配置持久化与设置页 UI
- 启动聊天时按配置创建本地 Mem0 记忆存储
- 与上述改动对应的测试

本设计不涵盖：

- 将主对话 LLM 改为本地运行
- 首期就暴露高级向量库或 embedding 调优选项
- 将历史云端记忆迁移或导入到本地

## 当前问题

### 设置窗口行为问题

`ChatWindow._open_settings()` 当前每次都会新建一个 `SettingsWindow`。结果是右键点一次就新增一个设置窗口，窗口数量没有上限，也破坏了“设置窗口应该只有一个”的使用预期。

### 记忆架构与需求不匹配

当前记忆初始化逻辑依赖 `MEM0_API_KEY` 并创建 `MemoryClient`，这更接近云端接入方式。现在的需求是：记忆数据必须本地化、轻量化，但主对话 LLM 可以继续使用远程服务，因此现有方案不合适。

## 推荐方案

采用独立的本地记忆配置，并将其与 API 配置分离。

推荐原因：

- 本地存储相关配置不应混入 API 页面
- 后续如果继续扩展记忆设置，不需要重新拆配置模型
- 避免把本地存储路径硬编码在启动逻辑里

## 设计

### 1. 设置窗口生命周期

从聊天窗口视角，将设置窗口改成单实例复用：

- `ChatWindow` 持有一个可复用的设置窗口引用，或者持有一个可返回共享设置窗口的回调
- 当用户点击“打开设置”时：
  - 如果设置窗口已经存在且仍然有效，则调用 `show()`、`raise()`、`activateWindow()`
  - 如果设置窗口不存在，则创建一个新的设置窗口，保存引用后再显示
- 当设置窗口被关闭时，需要清理保存的引用，这样下次才能重新创建

行为结果：

- 连续多次右键 -> “打开设置”不会产生重复窗口
- 已关闭的设置窗口仍然可以正常再次打开

### 2. 记忆配置模型

新增独立的记忆配置模型与配置文件。

配置字段：

- `enabled: bool`：是否启用本地记忆
- `storage_dir: str`：本地记忆持久化根目录
- `user_id: str`：用户记忆命名空间
- `top_k: int`：每次检索返回的最大记忆条数

持久化方式：

- 在 `config/schema.py` 中新增 `MemoryConfig`
- 在 `ConfigManager` 中新增读取与保存逻辑
- 配置文件落在 `data/config/memory.yaml`

默认值：

- `enabled = false`
- `storage_dir = "data/memory"`
- `user_id = "default-user"`
- `top_k = 5`

这些默认值满足“本地、简单、可预期”的原则。

### 3. 设置页 UI

在设置窗口中新增独立的“记忆”页面。

首期提供的控件：

- 启用本地记忆：复选框
- 存储目录：文本输入框
- 用户 ID：文本输入框
- 检索条数 `top_k`：数值输入控件

UI 行为：

- 页面风格保持与现有设置页一致
- 保存时通过 `ConfigManager` 持久化到 `memory.yaml`
- 页面中不出现云端 API Key 相关字段

### 4. 本地 Mem0 启动方式

将当前偏云端的记忆初始化逻辑替换为本地构造器。

启动行为：

- `SettingsWindow._launch_chat()` 读取 `self._config.memory`
- 如果记忆未启用，则向聊天窗口传入 `None`
- 如果记忆已启用：
  - 创建本地 Mem0 OSS 实例
  - 将其向量存储配置为 `storage_dir` 下的本地 Chroma 持久化目录
  - 用该实例包装出 `Mem0MemoryStore`

运行模型：

- 对话记忆的检索与写入都保留在本地磁盘
- 主对话生成仍然使用现有远程 LLM 配置
- 本地记忆目录成为正常应用数据目录的一部分，用户可以自行备份或清理

### 5. `Mem0MemoryStore` 职责

保持 `Mem0MemoryStore` 足够薄，只负责记忆接口适配：

- `search_relevant(query, user_id)` 返回纯文本记忆列表
- `add_conversation(user_text, assistant_text, user_id)` 写入一轮用户/助手消息
- 检索时遵守配置中的 `top_k`

它不负责：

- UI 逻辑
- 配置解析
- 聊天流程编排

### 6. 测试策略

按测试先行实现这次改动。

必须覆盖的测试：

- 设置窗口复用
  - 连续打开两次设置时应复用同一个窗口引用
  - 设置窗口关闭后再次打开时应创建新的窗口实例
- 配置管理
  - 能读取默认记忆配置
  - 能保存并重新加载 `memory.yaml`
- 本地记忆创建
  - 未启用记忆时返回 `None`
  - 启用记忆时能基于配置路径创建本地 Mem0 记忆存储
- 现有 agent 流程
  - 记忆检索仍然会注入到额外上下文中
  - 助手回复后仍然会回写记忆

## 错误处理

- 如果启用了本地记忆，但 Mem0 或其本地依赖不可用，启动时应向用户明确报错，而不是静默降级为无记忆
- 如果配置的存储目录非法，启动时应明确反馈路径错误
- 如果 `user_id` 为空，则在启动记忆前归一化为默认值

## 权衡

本次接受的权衡：

- 首期只暴露最小可用的记忆配置面，不提供高级 embedding、向量库调优等能力
- 本次保证的是“记忆数据本地持久化”，不是“整条记忆处理链完全离线”；主对话 LLM 继续保留远程方案，这是当前确认过的产品边界

## 实现边界

预期修改文件：

- `config/schema.py`
- `config/manager.py`
- `ui/settings/window.py`
- `ui/settings/pages/` 下新增记忆页面
- `ui/chat/window.py`
- `memory/mem0_store.py`
- 与配置、UI 行为、记忆创建相关的测试文件

预期新增文件：

- `data/config/memory.yaml` 可以选择运行时保存生成，不强制预置模板文件

## 成功标准

- 从聊天窗口右键打开设置时，不会出现重复的活动设置窗口
- 关闭设置窗口后可以再次正常打开
- 用户可以通过设置页启用本地记忆并指定持久化目录
- 记忆数据会持久化到本地配置目录
- 主对话仍然继续使用现有远程 LLM 配置
