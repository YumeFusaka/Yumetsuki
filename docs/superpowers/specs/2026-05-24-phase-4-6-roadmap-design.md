# Yumetsuki Phase 4-6 路线图设计

日期：2026-05-24

> 状态：草案已确认，可进入实施计划拆解。

## 背景

当前 Yumetsuki 已具备基础的：

- 角色聊天
- 工具调用
- MCP 接入
- 长期记忆
- 基础 TTS

但项目仍存在一组会直接影响桌宠核心体验的问题：

- 多轮对话连续性不足，缺少真正的单会话短期记忆
- 首字延迟不稳定，部分能力默认争抢首字前关键路径
- TTS 仍有无响应、卡死、分段并发失控等风险
- UI 仍偏向“功能壳”，桌宠感和在场感不足
- STT、视觉、浏览器自由操控等高级能力尚未完整落地

本设计文档将这些后续工作统一整理为 Phase 4、Phase 5、Phase 6 三个连续阶段，按“先稳核心链路，再扩桌宠体验，再扩高级代理能力”的顺序推进。

## 总体目标

- 让桌宠具备稳定的多轮连续对话能力
- 让首字、TTS、工具调用在长时间使用下保持可预期
- 让角色聊天界面从“聊天窗口”升级为更像桌宠的交互形态
- 完成语音输入输出闭环
- 最终扩展到浏览器自由操控与基础视觉能力

## 分阶段原则

- 先修基础链路稳定性，再做体验增强
- 先完成会话能力和线程治理，再接入更多输入输出设备与代理能力
- 长短期记忆职责分离：
  - `SessionContext` 负责短期记忆
  - `mem0` 继续负责长期记忆
- UI、STT、视觉、浏览器操控均建立在稳定会话层之上

## Phase 4：核心链路与会话能力

### 目标

把当前聊天主路径打磨成“不断线、不失忆、不假死”的稳定底座。

### 范围

- 首字延迟优化
- `SessionContext` 短期记忆系统
- 短期记忆存储：
  - 内存主存
  - SQLite 快照
- 长短期记忆分层：
  - `SessionContext` 管单会话工作记忆
  - `mem0` 管跨会话长期记忆
- 会话态内容：
  - recent turns
  - working facts
  - active tasks
  - session summary
- planner 拆为快路由与深规划
- 工具结果与任务状态回写短期会话态
- EventBus / 线程模型治理
- TTS 完善与稳定性修复
  - 超时
  - 取消
  - 并发上限
  - 固定流水线
  - 失败推进
- 第三阶段残留 TTS 功能补完

### 非目标

- 不在此阶段实现 STT
- 不在此阶段实现 OCR / 视觉
- 不在此阶段做浏览器自由操控
- 不在此阶段重做完整 UI 视觉语言

### 用户可见收益

- 上一句不会刚说完就忘
- 工具做过的事情下一轮还能接住
- 首字速度更稳定
- TTS 长时间运行不再容易卡死
- 普通聊天、工具聊天、多轮任务聊天的体验更一致

### 风险与依赖

- `SessionContext` 会影响 `AgentManager`、`LLMManager`、长期记忆写入策略，需要明确边界
- TTS 稳定性修复需要同时处理超时、顺序播放、并发和取消语义，不能只修表面症状
- EventBus 线程治理可能影响现有调试日志与设置页监听逻辑

### 验收标准

- 普通多轮聊天明显提升连续性
- 工具执行后的后续回复不再自相矛盾
- 长时间 TTS 使用不再出现明显卡死
- 首字延迟具备可测改善
- 短期记忆与长期记忆职责清晰，没有明显污染

## Phase 5：桌宠体验与交互输入输出

### 目标

让 Yumetsuki 从“能聊的工具程序”升级为“更像桌宠”的交互系统，并补齐语音输入。

### 范围

- 系统设置界面优化
- 角色聊天界面优化
- 系统设置支持：
  - 字体
  - 字号
  - 更多显示相关配置
- 被动互动模式
- 对话框隐藏后的气泡框文本显示
- 主动互动 / 被动互动状态设计
- STT 模块实现
- STT 接入聊天主链路
- STT / TTS 打断、切换和状态反馈

### 非目标

- 不在此阶段实现 OCR 视觉输入
- 不在此阶段完成浏览器高级代理能力
- 不在此阶段扩展 MCP / 插件生态的高级能力

### 用户可见收益

- UI 更像桌宠，而不是单纯聊天窗口
- 可以调节字体、字号等基础显示项
- 角色在非显式对话时也能以气泡形式表达存在感
- 语音输入可用于真实交互

### 风险与依赖

- 被动互动模式依赖 Phase 4 的稳定会话态，否则容易出现人格漂移或状态错乱
- STT 接入后会放大中断策略、短期记忆和任务连续性问题
- UI 优化需要在现有风格与桌宠交互之间找到平衡，避免只做表面美化

### 验收标准

- 设置页可修改字体、字号并立即体现在聊天界面
- 被动互动气泡稳定工作，不与主对话框状态冲突
- STT 能完成一轮从语音输入到回复输出的完整链路
- STT 与 TTS 不会频繁互相打断或锁死

## Phase 6：高级代理、插件生态与视觉

### 目标

让桌宠具备更完整的桌面代理能力，包括更强的浏览器操控、插件治理和视觉输入。

### 范围

- 插件管理界面优化
- MCP 管理能力增强
- 浏览器插件升级：
  - 更自由地操控浏览器
  - 更稳定的页面状态管理
  - 更强的浏览器交互抽象
- OCR 获取屏幕内容
- 视觉输入接入会话态
- 屏幕理解与工具链协同

### 非目标

- 不要求在此阶段一次性做到完整通用智能体
- 不要求在此阶段支持所有浏览器自动化场景
- 不要求视觉能力覆盖复杂多模态推理的全部需求

### 用户可见收益

- 插件与 MCP 更易管理、诊断与扩展
- 浏览器能力从“简单打开/搜索”升级为“持续操控”
- 桌宠能获取屏幕信息并基于视觉做出回应

### 风险与依赖

- 浏览器自由操控会显著提高状态管理复杂度
- OCR / 视觉输入会扩大上下文预算与隐私边界问题
- 插件与 MCP 管理界面若缺少清晰抽象，容易继续变成配置堆砌

### 验收标准

- 插件 / MCP 可清晰查看、启停与诊断
- 浏览器插件可执行复杂连续操作
- OCR 能稳定获取屏幕文本信息
- 视觉信息可进入后续对话或任务流程

## 阶段依赖关系

- Phase 4 是 Phase 5 与 Phase 6 的基础
- Phase 5 的被动互动和 STT 都依赖稳定会话态与线程治理
- Phase 6 的浏览器自由操控和视觉输入依赖更强的上下文管理能力

## 优先级结论

推荐固定顺序：

1. Phase 4
2. Phase 5
3. Phase 6

不建议调整该顺序。若在 Phase 4 未完成前推进 Phase 5 或 Phase 6，只会把现有链路问题放大到更复杂的交互场景中。

## 近期实施建议

在正式编码前，下一步应只拆 Phase 4 的实施计划，先明确：

- `SessionContext` 模块边界
- 短期记忆与 mem0 的衔接策略
- 首字前热路径与深能力层的时序边界
- TTS 稳定性与线程治理的最小改造路径

Phase 5 与 Phase 6 保持为已确认路线图，在 Phase 4 收敛后再各自展开详细实施计划。
