# Phase 5：桌宠体验、UI 与 STT 设计

日期：2026-05-24

> 状态：设计已确认，等待 Phase 4 收敛后进入详细实施。

## 背景

Phase 4 解决的是：

- 会话连续性
- 首字热路径
- TTS 稳定性
- 线程治理

Phase 5 要解决的是：

- 这个项目“看起来像不像桌宠”
- 输入输出链路是否自然

当前 UI 仍偏向工具窗口，缺少：

- 被动互动形态
- 气泡式存在感
- 用户可调字体 / 字号
- 真实语音输入

## 目标

- 优化系统设置与角色聊天界面
- 支持基础显示参数可调
- 增加被动互动展示模式
- 完成 STT 模块并接入主聊天链路

## 非目标

- 不在此阶段实现 OCR 视觉
- 不在此阶段实现浏览器高级代理能力
- 不在此阶段重做所有角色资源系统

## 设计原则

- UI 先服务桌宠存在感，再服务配置堆砌
- STT 必须建立在稳定会话态上
- 被动互动与主动聊天状态必须视觉上可区分
- 关键显示参数应配置化，而不是写死

## UI 优化范围

### 系统设置

建议增强：

- 字体家族
- 字号
- 聊天区与气泡区缩放
- 被动互动显示开关
- 被动互动显示位置 / 时间

### 聊天界面

建议优化：

- 对话框常态与被动互动形态切换
- 角色气泡框显示
- 被动互动时可隐藏主对话框
- 视觉反馈区分：
  - 用户主动发起
  - 角色主动提醒
  - 被动陪伴文本

## STT 设计范围

STT 在 Phase 5 中应至少支持：

- 语音输入转文本
- 文本进入正常聊天主链路
- 与 `SessionContext` 协同
- 与 TTS 的打断 / 切换逻辑

## 配置建议

以下关键参数应配置化：

- 字体
- 字号
- 气泡最大宽度
- 被动互动停留时长
- STT 引擎
- STT 录音超时
- STT 自动结束静音阈值

## 与 Phase 4 的依赖

Phase 5 必须建立在：

- `SessionContext`
- EventBus 线程治理
- 稳定的 TTS 管线

之上，否则：

- STT 会把短期记忆问题放大
- 被动互动会把线程问题和状态错乱放大

## 验收标准

- 系统设置可调整字体、字号并正确生效
- 被动互动模式可稳定显示气泡
- STT 可完成一轮语音输入到文本回复
- STT 与 TTS 不频繁互锁或互相误触发
