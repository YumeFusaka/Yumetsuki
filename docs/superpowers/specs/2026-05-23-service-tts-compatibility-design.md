# 服务端 TTS 对接规范文档设计

日期：2026-05-23

## 背景

Yumetsuki 当前已经与 GPT-SoVITS 形成一套稳定的桌宠端 TTS 对接方式，包括：

- `/tts` 合成请求
- `GET /set_refer_audio?refer_audio_path=...` 参考音频预热
- 参考模式切换（`auto` / `inline` / `session_preload` / `server_managed`）
- 原版路径兼容与自动回退
- 逐句请求、逐句翻译与逐句播放

近期围绕低延迟、`session_id` 会话化和 PCM 流式能力的讨论表明，项目需要一份**面向服务端开发者的长期规范文档**，明确什么可以改、什么不能改，以及 Yumetsuki 每次 TTS 侧变动后服务端应如何同步跟进。

当前已有文档的问题是：

- 信息分散在 `CLAUDE.md`、`docs/development.md` 和 `docs/superpowers/specs/2026-05-23-tts-pcm-design.md`
- 面向服务端的约束不够集中
- “原版兼容优先”的要求虽然已经写入项目文档，但缺少一份专门告诉服务端如何遵守该原则的专题文档

## 目标

新增一份正式专题文档，作为 Yumetsuki 服务端对接规范入口，服务于 GPT-SoVITS 及未来同类 TTS 服务端。

该文档必须做到：

- 明确原版兼容优先是最高原则
- 明确魔改只能通过调参、设置项和显式扩展字段实现，不能改动原版逻辑
- 明确 Yumetsuki 当前实际依赖的服务端行为
- 明确服务端允许做什么、禁止做什么
- 明确每次 Yumetsuki TTS 对接改动后，服务端应同步检查哪些内容
- 作为长期维护文档挂到 `docs/README.md` 入口中

## 非目标

- 不改动当前代码实现
- 不新增客户端功能
- 不把服务端设计稿直接并入这份规范文档
- 不替代 `docs/architecture.md` 中的代码结构说明
- 不替代 `docs/development.md` 中的本地开发流程说明

## 文档原则

这份规范文档应采用“规范书”写法，而不是“讨论稿”写法。

必须遵循以下表达原则：

- 使用强约束措辞：`必须`、`禁止`、`仅允许`、`不得`
- 不写模糊建议，如“可以考虑改默认行为”
- 不写站在服务端内部实现自由度角度的宽松表述
- 明确区分“原版路径”和“扩展路径”
- 明确标出“长期维护要求”

文档开头应单独强调一句核心规则：

> 魔改只能通过调参、设置项和显式扩展字段实现，不能改动原版逻辑。

## 文档结构设计

建议最终文档按以下结构组织。

### 1. 文档定位与适用范围

说明这是一份什么文档、给谁看、什么时候要更新。

应包含：

- 这是 Yumetsuki 面向服务端的正式对接规范
- 主要适用于 GPT-SoVITS 类 TTS 服务端
- 每次 Yumetsuki TTS 对接、兼容策略、请求参数、错误语义、回退逻辑、预热逻辑发生变化时，都必须同步更新该文档

### 2. 最高原则

明确优先级顺序：

1. 原版兼容优先
2. 魔改兼容次之
3. 桌宠端个性化适配再次之

并明确：

- 未显式启用扩展能力时，服务端必须保持原版行为不变
- 为桌宠端服务的扩展必须默认关闭或显式触发
- 不允许为了桌宠端而重写原版主逻辑

### 3. 服务端允许做什么

这一节只列**允许项**，避免服务端对“扩展”的理解过宽。

允许项包括：

- 调整原版已有参数的推荐值，但不得改默认值语义
- 增加可选设置项
- 增加可选扩展字段，如 `session_id`
- 增加向后兼容的响应头，如 PCM 格式元数据头
- 增加向后兼容的响应字段，如 `error_type`、`detail`
- 在显式扩展路径中增加会话能力、预热能力、PCM 流式能力

### 4. 服务端禁止做什么

这一节是文档重点之一，要写得比允许项更明确。

禁止项至少包括：

- 修改原版 `/tts` 默认 `media_type`
- 修改原版 `/tts` 默认 `streaming_mode`
- 改写原版 `/tts` 成功响应格式
- 改写原版错误响应中的基础 `message` 语义
- 把原版可选字段改成新必填项
- 改变原版 `GET /set_refer_audio?refer_audio_path=...` 的既有调用方式
- 因为支持桌宠端而替换原版逐次显式传参考字段的工作流
- 让无扩展字段请求落入魔改逻辑分支

### 5. Yumetsuki 当前依赖的服务端行为

这一节用于告诉服务端“客户端真的依赖什么”，避免无关扩展。

应覆盖：

- `/tts` 作为统一合成入口
- `GET /set_refer_audio?refer_audio_path=...` 预热路径
- 原版显式参考字段路径仍然可用
- 参考模式与回退模式并存
- 自动回退到逐次携带参考字段
- 常见语言别名兼容
- 句级请求与失败跳过语义

如果当前客户端对某能力只是“可利用”而不是“硬依赖”，文档中应明确标注为“扩展能力”而非“基础依赖”。

### 6. 服务端建议跟进方式

这一节说明服务端如何在不破坏原版的前提下提供 Yumetsuki 兼容能力。

应分成两条路径：

- 原版路径
  - 无 `session_id`
  - 保持原版默认行为
  - 保持显式参考字段工作流
- 扩展路径
  - 有 `session_id`
  - 可选会话预热
  - 可选 PCM 流式
  - 可选结构化错误与元数据头

核心要求：

- 扩展路径只能增量附着在原版路径之上
- 不能把扩展路径抬升为新的默认主线

### 7. 变更同步清单

这一节用于服务端开发者和 Yumetsuki 协作者在每次改动后快速核对。

至少应包括：

- 请求参数是否变化
- 默认值语义是否变化
- 新增字段是否为显式扩展
- 原版路径是否仍然兼容
- 参考音频预热方式是否变化
- 回退逻辑是否变化
- 错误语义是否变化
- 响应头是否新增或调整
- 是否需要新增兼容回归测试

### 8. 提交前检查清单

文档结尾给出 flat checklist，格式要适合直接照着过一遍。

建议包含：

- 原版 `/tts` 默认行为是否未变
- 原版 `/set_refer_audio` 调用方式是否未变
- 是否只新增扩展字段而未改旧字段语义
- 无 `session_id` 时是否仍兼容原版
- 显式启用扩展时是否仍可回退原版路径
- 新增错误字段时是否保留原版 `message`
- 是否补了原版兼容回归测试

## 维护规则

文档正文中需要明确写入以下维护要求：

- 每次 Yumetsuki 侧与服务端对接相关的改动，都必须同步更新该文档
- 该文档描述的是“当前项目对服务端的约束”，不是历史草稿
- 若某项客户端能力废弃或改语义，必须同步更新此文档，而不是只改代码

## 文案要求

为了避免服务端误解，文档中建议反复出现以下类型的表述：

- `必须保持原版默认行为不变`
- `仅允许作为显式扩展启用`
- `禁止为了兼容桌宠端而改写原版逻辑`
- `魔改只能调参和设置，不能改逻辑`

不应出现以下类型的表述：

- `服务端默认改成更适合桌宠端的行为`
- `为了低延迟可以直接替换原版默认路径`
- `只要客户端能用即可，不必保留原版调用方式`

## 风险与规避

- 风险：文档写成设计讨论，约束不够硬
  - 规避：统一使用规范性措辞，减少“建议性”语句
- 风险：把当前一次性的 TTS PCM 讨论内容混进长期规范
  - 规避：区分“长期原则”和“当前扩展能力示例”
- 风险：文档只写原则，不写服务端可执行检查项
  - 规避：加入提交前 checklist 与变更同步清单
