# 日志工作台打磨设计

日期：2026-05-24

> 状态：设计已确认，待拆实施计划。

## 背景

当前日志工作台基础能力已经落地，但在真实使用中仍有一批直接影响可读性、可排障性与可操作性的问题：

- 系统日志覆盖面不完整，记忆系统、LLM 输出切句、本地向 TTS 入队语句等关键链路不可见
- 系统日志页面上下两块区域会因自动刷新频繁重绘，导致滚动位置失控
- 系统日志来源筛选只有自由输入，用户很难理解其意义
- 对话日志把 `session_id` 直接暴露为会话入口，不符合普通用户心智
- 系统日志当前采用逐条列表，无法像普通文本一样直接用光标拖选复制多条日志
- 个别 UI 细节还不够成熟，例如系统日志级别下拉箭头样式、对话日志情绪标签样式

这些问题会直接削弱日志工作台作为 Phase 5 / 6 排障前置能力的价值，因此需要在现有结构上做一次聚焦打磨，而不是继续堆叠新入口。

## 目标

- 补齐记忆、LLM 切句与 TTS 入队等关键链路的系统日志覆盖
- 修复系统日志页自动刷新导致的滚动回弹和强制跟底问题
- 让系统日志既能结构化浏览，也能像普通文本一样自由选择复制
- 把来源筛选从“可输入但难理解”升级为“按业务链路 + 模块来源”的两层筛选
- 把对话日志中的会话入口改成更符合用户理解的“最近会话”
- 顺手收口当前日志页面的低质量 UI 细节

## 非目标

- 不重做 `LogEvent` 基础模型
- 不引入数据库、全文索引、分页或远程日志上传
- 不把系统日志页重做成复杂表格或监控平台
- 不在本次顺手重构整套 `LogService`

## 设计总览

本次改造遵循“保留现有结构、增强页面表现、补关键接线”的原则：

- 系统日志页改成“同源双视图”：
  - `结构化列表` 负责逐条浏览、定位和查看详情
  - `连续文本` 负责自由拖选、复制和全选
- 系统日志页的详情区继续保留 JSON 详情，但刷新策略从“每秒强制重写”改为“仅在选中事件变化时刷新”
- 对话日志页不再以手输 `session_id` 作为主入口，改为“最近会话下拉 + 当前会话 / 全部会话切换”
- 来源筛选改为两层：
  - 业务链路分组
  - 分组下的具体 `source`
- 在现有日志接线基础上补足：
  - 记忆检索与短期上下文摘要
  - LLM 输出切句摘要
  - 向 TTS 入队的句子与翻译摘要

## 系统日志页设计

### 页面结构

系统日志页维持上下布局，但上半区改为可切换视图：

- 顶部控制栏：
  - 业务链路分组
  - 具体来源筛选
  - 级别筛选
  - 关键字搜索
  - `仅当前会话`
  - `刷新`
  - `导出`
  - `打开目录`
  - `复制详情`
- 上半区：
  - `结构化列表` 视图
  - `连续文本` 视图
- 下半区：
  - 当前选中事件的 JSON 详情

两种上半区视图使用同一批筛选结果，不允许各自维护独立数据源。

### 双视图策略

#### 结构化列表

保留当前“每条事件一项”的浏览方式，继续承担：

- 快速扫读时间线
- 点选某一条日志
- 与下半区详情联动

该视图不承担自由复制多条文本的主要职责。

#### 连续文本

新增一个把当前筛选结果渲染为完整时间线文本的视图，要求：

- 支持鼠标拖选
- 支持系统复制与全选
- 保留完整时间、级别、来源、摘要与必要正文
- 与结构化列表视图共享筛选条件

该视图是“自由选择复制”的主入口。

### 自动刷新与滚动行为

系统日志页继续允许自动刷新，但必须改变刷新策略：

- 上半区若当前滚动位置已接近底部，刷新后允许继续跟底
- 只要用户已经主动上翻，刷新后必须保持当前位置
- 下半区详情文本在以下情况下不得自动重写：
  - 当前选中事件未变
  - 用户正在阅读或滚动详情

核心原则：

- 新日志追加不应打断阅读
- 自动刷新只追加结果，不应破坏用户当前视角

### 来源筛选模型

来源筛选改成两层。

第一层为业务链路分组，首版固定为：

- 全部
- 记忆
- LLM
- 切句
- TTS
- 工具
- UI
- Agent

第二层为具体来源模块，根据第一层动态收窄，例如：

- `TTS`
  - `chat.tts`
  - `tts.gptsovits`
- `LLM`
  - `llm.manager`
- `Agent`
  - `agent.manager`
- `记忆`
  - `session.manager`
  - 未来可扩展到 `memory.mem0`

分组映射首版采用静态维护，避免引入过度复杂的标签系统。

### 级别下拉样式

系统日志级别下拉框右侧箭头改为轻量化样式：

- 取消当前“色块感”明显的视觉效果
- 保留透明下拉背景
- 使用更细、更轻的小箭头或细线三角形
- 与现有樱花主题统一

本次仅调整样式策略，不改交互语义。

## 对话日志页设计

### 会话入口

对话日志页不再把“手输 `session_id`”作为默认入口，改为：

- `最近会话` 下拉框
- `当前会话`
- `全部会话`

最近会话下拉项应展示可读标签，而不是直接暴露纯 `session_id`。推荐格式：

- 最近时间
- 首句或首条摘要预览

`session_id` 仍保留为内部标识，但不再作为普通用户默认可见的主字段。

### 情绪标签样式

桌宠情绪标签继续采用内联胶囊，但视觉上改为：

- 宽度更舒展
- 左右留白更大
- 圆角更明显
- 与工具 / 记忆标签形成一致但更轻盈的视觉层级

目标是让情绪标签更像“状态胶囊”，而不是紧凑的小字章。

## 日志覆盖面补齐

## 1. 记忆系统

补充以下系统日志：

- 短期上下文构建摘要
- 长期记忆检索命中数
- 命中记忆预览摘要
- mem0 升格候选与跳过摘要

要求：

- 以摘要和计数为主
- 不默认写入过长原文
- 继续遵守脱敏规则

## 2. LLM 输出与切句

补充以下系统日志：

- 流式文本累计摘要
- 切句结果摘要
- 每次切出的待播报句子

这里的“切句”指客户端本地从 LLM 回复中切出 TTS 句段的过程，不要求记录每个字符级增量。

## 3. TTS 入队与翻译

补充以下系统日志：

- 向 TTS 入队的原始句子
- 是否需要翻译
- 翻译后文本摘要
- 入队、跳过、失败、超时、播放完成的完整链路

若句子较长，系统日志中仍以摘要方式呈现，并在详情里保留必要字段。

## 日志来源命名约束

为了支撑两层筛选，来源字段需要收敛为稳定集合。

首版要求：

- 新增日志尽量复用已有 `source`
- 同一链路内避免一个模块随意写出多个近义来源名
- 新增来源时必须同步纳入分组映射

来源分组是页面理解模型的一部分，不应完全依赖用户记忆具体模块名。

## 交互与复制策略

系统日志必须同时支持两种使用方式：

- 排障浏览：
  - 通过结构化列表逐条查看并点选详情
- 内容复制：
  - 通过连续文本视图直接用光标自由选择和复制

本次不以“复制当前筛选结果按钮”替代光标选择，因为用户目标是像普通文本一样直接操作。

## 测试要求

### 系统日志页

- 自动刷新时，非底部位置不应被强制滚到底
- 自动刷新时，底部附近允许继续跟底
- 详情区刷新不应重置滚动位置
- 连续文本视图应正确渲染当前筛选结果
- 业务分组与具体来源联动正确

### 对话日志页

- 最近会话下拉应渲染可读标签
- 当前会话 / 全部会话切换正确
- 情绪标签 HTML 应输出新胶囊样式

### 日志接线

- 记忆检索摘要事件存在
- 短期上下文摘要事件存在
- LLM 本地切句事件存在
- TTS 入队与翻译摘要事件存在

## 验收标准

- 系统日志覆盖面明显补齐，记忆、切句、TTS 入队链路可见
- 系统日志上半区不会再因自动刷新强制滚到底
- 系统日志下半区不会再因自动刷新回弹到顶部
- 系统日志支持连续文本视图，可直接用光标自由选择复制
- 来源筛选具备“业务链路 + 具体来源”两层含义
- 对话日志默认不再暴露难懂的会话码入口
- 系统日志级别下拉箭头视觉质量提升
- 对话日志情绪标签视觉更舒展
