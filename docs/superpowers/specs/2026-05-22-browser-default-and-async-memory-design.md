# 浏览器默认行为与异步记忆写入设计

## 背景

当前项目存在三个用户可感知的问题：

1. `打开浏览器` 没有真正使用系统默认浏览器，而是硬编码优先打开 Edge/Chrome/Firefox。
2. LLM 文本流式回复结束后，输入框仍会短暂不可用，影响连续对话体验。
3. 用户先让角色打开浏览器，再要求“用浏览器搜索 xxx”时，系统会再拉起一个新的 Playwright 可见浏览器实例，而不是沿用系统默认浏览器的既有会话；该新窗口还可能短暂显示后关闭，体验不符合预期。

本设计目标是修正上述行为，使“像人一样用电脑”的浏览器操作走系统默认浏览器，使后台自动化能力与普通浏览器打开行为分离，并消除对话结束后的额外阻塞。

## 目标

- `open_browser()` 真正打开系统默认浏览器。
- “用浏览器搜索 xxx”默认走系统默认浏览器搜索，而不是 Playwright 可见浏览器。
- `memory_store.add_conversation()` 改为异步执行，不阻塞用户下一轮发送。
- 保留 `web_automation` 作为后台网页自动化能力，用于搜索结果提取、网页文本提取、网页截图等任务。
- 通过测试覆盖路由和非阻塞行为，避免回归。

## 非目标

- 本次不实现对系统默认浏览器窗口或标签页的精确控制。
- 本次不尝试把 Playwright 会话附着到用户已手动打开的浏览器进程。
- 本次不新增浏览器自动点击、滚动、表单填写等复杂自动化能力。
- 本次不实现完整的记忆写入任务队列，仅先将对话记忆写入移到后台线程。

## 方案总览

浏览器相关能力分为两类：

1. 系统默认浏览器路径
   - 面向“打开浏览器”“用浏览器搜索”“在浏览器打开网址”这类桌面使用场景。
   - 行为目标是使用操作系统默认关联，尽量复用用户当前浏览器环境，而不是启动独立自动化实例。

2. Web 自动化路径
   - 面向“后台搜索并返回结果”“提取网页正文”“截图网页”等程序性操作。
   - 继续使用 Playwright。
   - `web_search_visible` 不再作为普通“浏览器搜索”的默认选择，只在明确需要展示自动化搜索过程时才使用。

同时，对话结束后的记忆写入改为后台执行，主回复链只负责尽快结束流式输出并释放 UI 发送锁。

## 详细设计

### 1. 系统默认浏览器打开

修改 `plugins/system_control/open.py`：

- `do_open_browser()` 不再手动解析 Edge/Chrome/Firefox 路径。
- 改为通过系统默认关联打开浏览器。
- 在 Windows 上使用 `os.startfile()` 打开一个稳定的浏览器地址，例如搜索引擎首页。

设计约束：

- 不再把“默认浏览器”建立在“常见浏览器可执行文件探测”之上。
- 返回结果文案必须与实际行为一致。

兼容性选择：

- `about:blank` 在不同环境下兼容性不如标准 HTTP(S) 地址稳定。
- 因此使用一个明确的 HTTP(S) 首页地址更稳妥。

### 2. 默认浏览器搜索能力

在系统控制插件中新增“默认浏览器搜索”能力。

候选实现方式：

- 方案 A：新增 `search_in_browser(query, engine="")`
- 方案 B：扩展 `open_url()` 由 Planner 拼搜索 URL

采用方案 A，原因如下：

- 工具语义更清晰，LLM 更容易稳定选对。
- 可以把搜索引擎选择逻辑封装在插件内，而不是散落到 Planner 参数拼接逻辑。
- 便于后续继续加“默认搜索引擎配置”而不改 Planner。

行为定义：

- 用户说“用浏览器搜索 xxx”“打开浏览器搜索 xxx”“在浏览器里搜 xxx”时，优先调用该工具。
- 工具内部根据配置或参数决定搜索引擎，拼接搜索 URL 后交给系统默认浏览器打开。
- 返回值仅确认已发起搜索，不承诺抓取页面文本内容。

### 3. Planner 路由规则调整

修改 `agent/planner.py` 中的 LLM judge 规则和相关工具描述，使浏览器意图有明确边界：

- “打开浏览器”“用浏览器搜索”“在浏览器中打开 URL”优先选择系统控制插件中的默认浏览器工具。
- “后台搜索并返回结果”“提取网页内容”“网页截图”优先选择 `web_automation`。
- “显示自动化搜索过程”“可见浏览器自动执行搜索步骤”才允许优先使用 `web_search_visible`。

路由原则：

- 优先匹配能直接满足用户桌面操作预期的工具。
- 只有当用户明确要求“返回搜索结果内容”或“自动化处理网页”时，才进入 Playwright 路径。

### 4. Web 自动化工具描述修正

调整 `plugins/web_automation/plugin.py` 中工具说明文字，降低 LLM 误选概率：

- `web_search`
  - 强调它是后台静默搜索。
  - 强调用户不会看到浏览器。
  - 强调适用于“只要搜索结果文本摘要”的场景。

- `web_search_visible`
  - 强调它会启动受 Playwright 控制的自动化浏览器。
  - 强调它不等同于系统默认浏览器的现有窗口。
  - 强调它仅适用于用户明确要求看到自动化搜索过程时。

本次不删除 `web_search_visible`，因为它仍有调试和演示价值，但不应成为普通浏览器搜索的首选。

### 5. 异步对话记忆写入

修改 `agent/manager.py`：

当前流程为：

1. LLM 流式输出完成
2. 同步执行 `memory_store.add_conversation(...)`
3. 启动异步反思线程
4. `chat_stream()` 返回，UI 才解除发送锁

问题在于第 2 步可能很慢，尤其是本地向量模型或存储层较慢时，会直接拖慢 UI 解锁。

调整后的流程为：

1. LLM 流式输出完成
2. 立即结束主生成流程，让 UI 尽快收到完成信号
3. 后台线程异步执行：
   - `memory_store.add_conversation(...)`
   - 反思与深层记忆提取

实现方式：

- 把“对话记忆写入 + 反思”的后台逻辑统一纳入一个后台线程中顺序执行。
- 这样可以保留“先写入本轮对话，再做反思提取”的相对顺序，同时不阻塞主线程。

错误处理：

- 任一后台步骤失败时，不影响当前轮对话完成状态。
- 失败信息应通过事件总线日志或最小化日志记录暴露给调试面板，而不是反馈为前台对话错误。

### 6. 一致性与权衡

异步化带来的已知权衡：

- 用户如果在极短时间内连续发送两条消息，第二条消息的记忆检索未必能立即看到第一条刚写入的对话记忆。

本次接受该权衡，理由：

- 对话流畅性优先于强一致记忆。
- 当前问题是明显的交互阻塞，必须先修复。

后续可演进方向：

- 如果未来需要更强一致性，可以引入单独的后台写入队列和状态追踪，但不作为本次范围。

## 测试策略

### 单元测试

1. 系统默认浏览器打开
   - mock `os.startfile`
   - 验证 `do_open_browser()` 不再解析固定浏览器路径，而是通过默认关联打开 URL

2. 默认浏览器搜索
   - 验证搜索 URL 拼接正确
   - 验证工具调用最终使用默认浏览器打开 URL

3. Planner 路由
   - 对“用浏览器搜索 Python 教程”之类输入做回归测试
   - 验证优先路由到默认浏览器搜索工具，而不是 `web_search_visible`

4. 异步记忆写入
   - 构造一个 `add_conversation()` 明显延迟的假 memory store
   - 验证 `chat_stream()` 能在不等待该延迟结束的情况下完成主流程

### 验证性测试

- 运行目标测试文件
- 执行 `python -m pytest tests/ -q`
- 对涉及文件运行 `py_compile` 或等效基础语法校验

## 验收标准

- 用户说“打开浏览器”时，打开的是系统默认浏览器，而不是硬编码指定浏览器。
- 用户说“用浏览器搜索 xxx”时，不再弹出新的 Playwright 自动化窗口，而是走系统默认浏览器搜索。
- 回复文本结束后，输入框应立即恢复可发送状态，不再因记忆写入产生额外停顿。
- 后台网页搜索摘要、网页文本提取、网页截图能力保持可用。
- 新增或修改的测试通过。
