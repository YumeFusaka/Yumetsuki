# Yumetsuki UI 规范

## 设计方向

Yumetsuki 的主 UI 是 Tauri / Vue。界面必须保持桌宠角色感、Sakura 浅色主题、轻盈但克制的视觉语言，以及对聊天、设置、日志和诊断的高频操作效率。

历史 Qt UI 已退场。本文只描述当前 Vue / Sakura 组件体系，不再保留 Qt 控件规范。

## 颜色与基调

- 主背景保持浅粉、白和柔和玫瑰色层次。
- 强调色使用深玫瑰红，不使用高饱和紫蓝渐变做主视觉。
- 页面整体保持安静、易扫描、偏工作台而不是营销页。

## 组件原则

- 页面 sections 使用全宽布局，不把页面本体包成卡片。
- 卡片只用于重复项、对话框或确实需要边界的局部控件。
- 按钮、开关、下拉、分段控件、滑块和输入框分别使用对应语义组件。
- 图标按钮优先使用图标语义，不用装饰性圆角文字块代替。
- 所有控件要保证文本不溢出、不重叠，移动端和桌面端都能稳定换行或缩放。

## 页面规范

### 主壳

- 左侧导航承载聊天、设置、日志、工具和诊断入口。
- 当前页状态要清晰，但不要把导航做成浮夸卡片。
- 启动状态区域展示 sidecar 就绪、降级或异常信息。

### 聊天页

- 聊天正文区域必须可滚动，长文本不得撑破布局。
- 发送、停止、重试、语音输入和运行状态必须在同一工作区内可达。
- 流式回复应避免每个 chunk 都触发完整重绘。
- 被动状态使用独立气泡容器，但仍属于普通聊天链路。

### 设置页

- 设置项按 API、角色、记忆、Agent、插件、MCP、日志和系统分组组织。
- 保存语义要明确区分 API / 系统等不同作用域。
- 未保存修改应有清晰的脏状态提示。

### 日志页

- 对话日志和平台日志必须支持筛选、复制、自动刷新和暂停刷新。
- 平台日志结构化列表应保持紧凑，详情区承载完整 JSON。
- 刷新时不得打断用户选择文本或滚动位置。

### 诊断页

- 运行状态、进度、导出和失败态必须明确。
- 敏感扫描失败时应阻止导出，并给出可理解的错误摘要。

## 交互反馈

- 成功和失败优先使用轻量 toast。
- 高风险操作使用确认对话框。
- 忙碌态要能区分“生成中”“停止中”“失败”“可重试”等状态。
- 所有反馈文案应短、准、可执行，不写教学式说明。

## 禁止项

- 黑底系统弹窗样式直接暴露。
- 页面主体卡片化过度。
- 用文字块伪装图标。
- 让单个视图同时承载过多职责。
- 重新引入 Qt / PySide6 控件规范作为当前 UI 约束。
