# 服务端 TTS 对接规范

> 最后更新：2026-05-23
>
> 核心规则：魔改只能通过调参、设置项和显式扩展字段实现，不能改动原版逻辑。

## 文档定位

本文档是 Yumetsuki 面向服务端开发者的正式对接规范，适用于 GPT-SoVITS 及其他需要跟随 Yumetsuki TTS 对接方式演进的服务端。

- 本文档必须作为服务端兼容性约束的长期维护入口。
- 本文档必须界定服务端允许变更的范围、禁止变更的范围，以及 Yumetsuki 当前实际依赖的服务端行为。
- 本文档不得替代服务端内部设计文档；服务端内部实现可以调整，但对外协议与默认行为必须遵守本文档。
- 每次 Yumetsuki 的 TTS 对接方式、请求参数、错误语义、回退逻辑、预热逻辑或扩展能力协商发生变化时，必须同步更新本文档。
- 本文档描述的是当前有效规范，不是讨论稿、路线图或一次性设计记录。

## 术语

### 原版

- `原版` 指服务端上游项目已经公开提供并稳定约定的接口行为、默认参数语义、响应结构、错误语义和调用方式。
- `原版` 不等于“当前服务端维护者个人常用配置”；凡是未在上游原始协议中显式声明为新增扩展的能力，均不得被视为原版的一部分。

### 原版路径

- `原版路径` 指请求未显式进入扩展协商时，服务端必须执行的默认处理主线。
- 原版路径必须保持原版 `/tts` 默认行为、原版成功响应结构、原版基础错误语义，以及原版显式参考字段工作流。

### 扩展路径

- `扩展路径` 指请求已经满足显式扩展触发条件后，服务端才允许启用的附加处理主线。
- 扩展路径必须附着在原版路径之上，不得替代原版路径，不得改变未启用扩展时的默认行为。

### 显式扩展字段

- `显式扩展字段` 指服务端文档已经明确标注为扩展用途，且客户端必须主动携带才能触发扩展路径的字段或协商项。
- `session_id` 属于显式扩展字段。
- 服务端自定义的扩展请求头、扩展查询参数、扩展请求体字段，只有在文档中被明确定义为扩展协商项时，才允许视为显式扩展字段。
- 未文档化的未知字段不得视为显式扩展字段。

### 桌宠端专用扩展

- `桌宠端专用扩展` 指仅为 Yumetsuki 或同类桌宠客户端增加的兼容能力。
- 本文中的 `魔改` 仅指桌宠端专用扩展，不指对原版逻辑的重写。
- 桌宠端专用扩展必须通过调参、设置项和显式扩展字段实现，不得改动原版逻辑。

## 最高原则

### 原版兼容优先

- 服务端必须先保证原版接口、默认参数、返回格式、错误语义和基础调用方式不变。
- 未显式启用扩展能力时，服务端必须保持原版行为不变。
- 服务端仅允许在不破坏原版路径的前提下增加 Yumetsuki 兼容能力。
- 服务端不得为了桌宠端适配而重写原版默认主线。

### 魔改只能调参和设置，不能改逻辑

- 魔改只能通过调参、设置项和显式扩展字段实现，不能改动原版逻辑。
- 服务端不得为了桌宠端专用扩展而改写原版主流程、原版默认值、原版必填项或原版返回结构。
- 桌宠端专用扩展仅允许放在显式扩展路径中处理，不得让原版请求自动落入扩展逻辑。

### 扩展路径必须显式触发

- 服务端仅允许在请求满足已文档化的显式扩展触发条件时进入扩展路径。
- 未知字段、拼写错误字段、附带调试字段或仅出现原版字段的请求，不得自动切换主线。
- 单独出现 `ref_audio_path`、`prompt_lang`、`prompt_text`、`text_lang`、`streaming_mode` 或其他原版字段时，必须仍按原版语义解释。

## 服务端允许做什么

- 服务端仅允许通过调参优化推荐参数组合，但不得改写原版默认参数语义。
- 服务端仅允许新增可选设置项，不得借新增设置项废弃原版默认路径。
- 服务端仅允许新增可选显式扩展字段，例如 `session_id`。
- 服务端仅允许新增向后兼容的响应头和响应字段，例如扩展路径的元数据头、`error_type`、`detail`。
- 服务端仅允许在扩展路径中增加会话预热、服务端托管参考、PCM 流式和结构化错误能力。
- 服务端仅允许在请求已经进入扩展路径后，对缺省扩展参数做补全。
- 服务端仅允许新增兼容性检测、诊断信息和回退提示，但不得改变原版成功 / 失败语义。

## 服务端禁止做什么

- 禁止修改原版 `/tts` 默认 `media_type`。
- 禁止修改原版 `/tts` 默认 `streaming_mode`。
- 禁止改写原版 `/tts` 成功响应格式。
- 禁止破坏原版错误响应中的基础 `message` 语义。
- 禁止把原版可选字段改成新的必填项。
- 禁止改变原版 `GET /set_refer_audio?refer_audio_path=...` 的既有调用方式。
- 禁止因为支持桌宠端而替换原版逐次显式传参考字段的工作流。
- 禁止让无扩展字段请求落入扩展逻辑分支。
- 禁止为了低延迟、会话化或流式播放而重写原版默认主线。
- 禁止要求 Yumetsuki 在未显式启用扩展时必须携带 `session_id` 或其他桌宠端专用字段。
- 禁止通过修改原版返回结构来隐式绑定 Yumetsuki 客户端。
- 禁止在扩展失败时移除原版回退路径。
- 禁止把未文档化字段、未知字段或仅供内部调试的字段解释为进入扩展路径的依据。

## Yumetsuki 当前依赖的服务端行为

### 基础依赖

- `/tts` 必须作为统一合成入口。
- `GET /set_refer_audio?refer_audio_path=...` 必须作为当前已支持的参考音频预热路径。
- 原版逐次显式传参考字段的工作流必须保持可用；Yumetsuki 在 `inline` 和 `auto` 回退场景中仍会依赖该路径。
- 服务端必须接受 `text` 与 `text_lang` 组成的基础合成请求；在携带参考字段时，还必须保持对 `ref_audio_path`、`prompt_lang`、`prompt_text` 等原版显式参考信息的兼容。
- 服务端必须允许句级请求、单句失败跳过和顺序播放语义共存，不得要求整段文本一次性成功才返回可用结果。
- 服务端必须兼容常见语言别名归一化后的结果，例如 `jp -> ja`。

### 扩展能力边界

- `session_preload`、`server_managed` 与 `auto` 属于 Yumetsuki 可利用的兼容策略；服务端若实现相关能力，必须保持原版路径仍可回退。
- `auto` 模式下，若预热失败或服务端仍要求显式参考字段，Yumetsuki 会回退到逐次携带参考字段；服务端不得阻断该回退。
- `session_id`、PCM 流式、结构化错误与元数据头属于扩展能力，不属于 Yumetsuki 当前的无条件基础依赖。
- 服务端若支持会话能力，只能把 `session_id` 作为可选扩展键，不得把它提升为原版路径前置条件。

### 客户端模式边界

- Yumetsuki 将 `audio_mode=wav + reference_mode=inline` 定义为保底模式。
- 在该模式下，客户端不得透传 `session_id`，不得调用 `set_refer_audio`，不得发送 `media_type=raw`、`streaming_mode=3` 或其他桌宠端扩展参数。
- `pcm_stream + inline` 属于音频扩展：客户端允许显式请求流式音频，但不应期待当前服务端实现的会话扩展语义。
- 只有显式带 `session_id` 的组合，才属于当前服务端实现里的会话扩展。

### 客户端模式总表

术语约定：

- `保底模式`：桌宠端强制只走原版显式参考字段工作流
- `音频扩展`：只扩展音频返回方式，例如 PCM 流式
- `会话扩展`：显式带 `session_id`，进入当前服务端实现的会话化参考路径

#### 参考模式视角

| 参考模式 | 客户端是否每次带显式参考 | 是否依赖 `set_refer_audio` | 是否依赖 `session_id` | 谁主导参考状态 | 兼容性定位 | 当前实测结果 | 典型失败后果 |
|---|---|---|---|---|---|---|---|
| `inline` | 是 | 否 | 否；保底模式下明确禁用 | 客户端 | 原版基线 / 最保守 | 当前仅 `wav + inline` 可确认完全通过；若与 PCM 组合，仍有问题待处理 | 单次请求失败，但不依赖会话状态 |
| `session_preload` | 通常首轮预热后尽量不带；失败时回退显式参考 | 是 | 是，属于核心扩展参数 | 客户端先灌入，服务端会话复用 | 会话扩展 | 已能跑通部分链路，但仍有扩展协商、warmup 或播放问题待处理 | 预热失败后回退到 `inline` |
| `server_managed` | 通常不带 | 否或不依赖客户端主动预热 | 常见会依赖，但取决于服务端设计 | 服务端 | 会话扩展 / 最依赖服务端 | 目前未确认完全通过，仍需继续联调验证 | 服务端参考未就绪时整体不可用 |
| `auto` | 由客户端按探测结果决定 | 可能会 | 可能会 | 混合 | 客户端策略项 | 已具备回退链路，但整体仍未达到“完全无问题”；尤其 PCM 分支仍有待继续处理 | 失败后回退到更保守路径 |

#### 音频模式 + 参考模式组合视角

| 音频模式 | 参考模式 | 当前定位 | 是否允许 `session_id` | 是否允许 `set_refer_audio` | 当前实测结果 | 说明 |
|---|---|---|---|---|---|---|
| `wav` | `inline` | 保底模式 | 否 | 否 | 完全通过，可作为当前唯一稳定保底组合 | 强制原版路径，互不侵犯的保底模式 |
| `wav` | `session_preload` | 会话扩展 | 是 | 是 | 部分通过，仍需继续联调验证 | 不流式，但走会话参考扩展 |
| `wav` | `server_managed` | 会话扩展 | 允许 | 通常不需要 | 目前未确认完全通过 | 参考完全交给服务端 |
| `wav` | `auto` | 客户端策略项 | 可能 | 可能 | 部分通过，但仍不作为当前稳定保底组合 | 客户端探测参考策略 |
| `pcm_stream` | `inline` | 音频扩展 | 否 | 否 | 存在问题，仍需继续处理 | 只扩展音频返回方式，不扩展参考会话 |
| `pcm_stream` | `session_preload` | 音频扩展 + 会话扩展 | 是 | 是 | 存在较多问题，需继续处理 | 当前最完整的低延迟会话方案 |
| `pcm_stream` | `server_managed` | 音频扩展 + 会话扩展 | 允许 | 通常不需要 | 目前未确认稳定 | 最依赖服务端实现 |
| `pcm_stream` | `auto` | 客户端策略项 | 可能 | 可能 | 问题最多，当前不应视为稳定方案 | 低延迟优先策略 |
| `auto` | 任意非 `wav + inline` 组合 | 客户端策略项 | 可能 | 可能 | 仍在演进中，不保证完全稳定 | 会先尝试扩展能力，再按策略回退 |

### 当前问题归因边界

- 当 Yumetsuki 已经在显式扩展路径下正确传递 `session_id`、`prompt_lang`、`prompt_text` 后，服务端 warmup 内部生成何种探测文本、以及该探测文本是否与 `prompt_lang` 对齐，属于服务端责任。
- 当服务端日志或实际输出出现 `、。` 之类由服务端内部切句、补句号或 warmup 产物带来的异常时，不应把该异常反向归因到 Yumetsuki 当前桌宠端切句逻辑，除非客户端日志已经证明桌宠端本地先产出了相同异常分段。

## 服务端建议跟进方式（原版路径 / 扩展路径）

### 原版路径

- 无显式扩展字段、无显式扩展请求头、无显式扩展查询参数时，服务端必须完全按原版路径处理。
- 服务端必须保持 `/tts` 的原版默认行为、原版返回格式与原版错误语义。
- 服务端必须保持逐次显式传参考字段的工作流。
- 服务端必须保持 `GET /set_refer_audio?refer_audio_path=...` 的原版调用方式可用。
- 服务端不得在原版路径中混入桌宠端专用默认补全逻辑。

### 扩展路径

- 扩展路径仅允许在请求满足显式扩展触发条件时启用。
- `session_id` 仅允许作为可选会话键，不得成为原版请求前置条件。
- 会话预热、服务端托管参考、PCM 流式、结构化错误、附加元数据头仅允许作为扩展能力附着在原版路径之上。
- 扩展路径必须保留回退到原版路径的能力；扩展协商失败、预热失败或能力未开启时，不得阻断原版合成。
- 服务端不得把扩展路径抬升为新的默认主线。

### 扩展路径触发条件

- 请求体、查询参数或请求头中必须出现至少一个已文档化的显式扩展字段或显式扩展协商项，服务端才允许进入扩展路径。
- `session_id` 明确属于可触发扩展路径的显式扩展字段。
- 在 `GET /set_refer_audio?refer_audio_path=...` 上显式携带已文档化的 `session_id`，仅允许触发“会话预热扩展”；不得因此改变 `/tts` 原版默认语义。
- 若服务端把 `prompt_lang`、`prompt_text` 也文档化为该会话预热扩展的必填伴随字段，客户端仅允许在显式扩展路径下与 `session_id` 一起携带；缺失这些扩展字段时，客户端应退回原版预热或逐次显式参考字段路径。
- 若服务端定义了其他扩展协商项，例如扩展请求头或扩展字段，必须在服务端文档中明确写出“该字段仅用于扩展路径”，否则不得作为触发条件。
- 未知字段、未文档化字段、仅出现原版字段、仅出现内部调试字段时，服务端不得自动进入扩展路径。
- 单独出现 `ref_audio_path`、`prompt_lang`、`prompt_text` 或原版 `streaming_mode` 时，服务端不得据此判定客户端要求桌宠端专用扩展。

### 原版路径示例

以下请求必须按原版路径处理，不得自动切换为扩展路径：

```http
POST /tts
Content-Type: application/json

{
  "text": "你好。",
  "text_lang": "zh",
  "ref_audio_path": "ref.wav",
  "prompt_lang": "zh",
  "prompt_text": "示例参考文本"
}
```

判定说明：

- 请求中只包含原版字段。
- 即使服务端内部已经实现 `session_id`、PCM 流式或其他桌宠端专用扩展，也不得在本请求上默认启用。
- 若服务端支持预热接口，以下调用也仍属于原版路径，不得据此改变 `/tts` 默认语义：

```http
GET /set_refer_audio?refer_audio_path=ref.wav
```

### 扩展路径示例

以下请求满足显式扩展触发条件，服务端才允许进入扩展路径：

```http
GET /set_refer_audio?refer_audio_path=ref.wav&session_id=sess-123
```

```http
GET /set_refer_audio?refer_audio_path=ref.wav&session_id=sess-123&prompt_lang=zh&prompt_text=%E7%A4%BA%E4%BE%8B
```

```http
POST /tts
Content-Type: application/json

{
  "text": "你好。",
  "text_lang": "zh",
  "session_id": "sess-123"
}
```

判定说明：

- `session_id` 是已文档化的显式扩展字段，因此服务端允许进入会话化扩展路径。
- 若服务端要求 `prompt_lang`、`prompt_text` 作为该扩展路径的伴随字段，客户端应在该扩展请求中一并显式携带；未携带时，客户端不应把失败误判为原版路径不可用。
- 若 `session_id` 对应的参考信息尚未准备好，服务端必须保持原版基础错误语义，并且仅允许在此基础上附加扩展错误字段。
- 若请求只携带未知字段，或者只携带 `prompt_lang` 等原版字段而没有任何显式扩展字段，服务端不得按本示例进入扩展路径。

## 变更同步要求

- 每次 Yumetsuki 涉及 TTS 对接、兼容策略、请求参数、默认值语义、错误语义、回退逻辑、预热逻辑、响应头或扩展字段的改动，都必须同步更新本文档。
- 每次服务端准备提交与 Yumetsuki 兼容相关的改动时，必须先对照本文档检查是否破坏原版路径。
- 若某项客户端能力被废弃、替换或改语义，必须同步更新本文档，不得只改代码。
- 若服务端新增扩展能力，必须明确标注对应的显式扩展字段或显式扩展协商条件。
- 若新增能力无法用“原版路径不变、扩展路径显式触发”解释，该能力不得直接合入服务端默认主线。
- 本文档描述的是当前项目对服务端的约束，不是历史草稿；任何过期设计稿都不得替代本文档。

## 服务端提交前检查清单

- 原版 `/tts` 默认行为是否未变。
- 原版 `/set_refer_audio` 调用方式是否未变。
- 是否已经为新增扩展能力定义显式扩展字段或显式扩展协商条件。
- 是否只新增扩展字段而未改旧字段语义。
- 无 `session_id` 时是否仍兼容原版。
