# 服务端 TTS 对接规范

> 最后更新：2026-05-23
>
> 核心规则：魔改只能通过调参、设置项和显式扩展字段实现，不能改动原版逻辑。

## 文档定位

本文档是 Yumetsuki 面向服务端开发者的正式对接规范，适用于 GPT-SoVITS 及其他需要跟随 Yumetsuki TTS 对接方式演进的服务端。

- 本文档必须作为服务端兼容性约束的长期维护入口。
- 本文档必须用于界定服务端允许变更的范围、禁止变更的范围，以及 Yumetsuki 当前实际依赖的服务端行为。
- 本文档不得替代服务端内部实现设计文档；服务端内部如何实现由服务端自行决定，但对外行为必须遵守本文档。
- 每次 Yumetsuki 的 TTS 对接方式、请求参数、错误语义、回退逻辑、预热逻辑或扩展能力协商发生变化时，必须同步更新本文档。
- 本文档描述的是当前有效规范，不是讨论稿、路线图或一次性设计记录。

## 最高原则

### 原版兼容优先

- 服务端必须先保证原版接口、默认参数、返回格式、错误语义和基础调用方式不变。
- 未显式启用扩展能力时，服务端必须保持原版行为不变。
- 服务端仅允许在不破坏原版路径的前提下增加 Yumetsuki 兼容能力。
- 服务端不得为了桌宠端适配而重写原版默认主线。

### 魔改只能调参和设置，不能改逻辑

- 魔改只能通过调参、设置项和显式扩展字段实现，不能改动原版逻辑。
- 禁止为了兼容桌宠端而改写原版主流程、原版默认值、原版必填项或原版返回结构。
- 仅允许把桌宠端专用能力放在显式扩展路径，不得让原版请求自动进入魔改逻辑。
- 服务端不得把“兼容 Yumetsuki”解释为“可以替换原版行为”。

### 扩展必须显式启用

- `session_id`、会话预热、服务端托管参考、PCM 流式、结构化错误、附加响应头等能力，必须作为显式扩展能力处理。
- 扩展路径不得抬升为新的默认主线。
- 未显式启用扩展时，服务端必须继续兼容原版 `/tts` 与原版显式参考字段工作流。
- 魔改只能通过调参、设置项和显式扩展字段实现，不能改动原版逻辑；该规则必须覆盖所有新增扩展能力。

## 服务端允许做什么

- 服务端仅允许通过调参优化推荐参数组合，但不得改写原版默认参数语义。
- 服务端仅允许新增可选设置项，不得借新增设置项废弃原版默认路径。
- 服务端仅允许新增可选扩展字段，例如 `session_id` 或其他显式协商字段。
- 服务端仅允许新增向后兼容的响应头和响应字段，例如扩展路径的元数据头、`error_type`、`detail`。
- 服务端仅允许在显式扩展路径中增加会话预热、服务端托管参考、PCM 流式和结构化错误能力。
- 服务端仅允许在请求已经显式进入扩展路径时，对缺省扩展参数做补全。
- 服务端仅允许新增兼容性检测、回退提示或诊断信息，不得改变原版成功 / 失败语义。
- 魔改只能通过调参、设置项和显式扩展字段实现，不能改动原版逻辑；任何允许项都不得突破这条边界。

## 服务端禁止做什么

- 禁止修改原版 `/tts` 默认 `media_type`。
- 禁止修改原版 `/tts` 默认 `streaming_mode`。
- 禁止改写原版 `/tts` 成功响应格式。
- 禁止破坏原版错误响应中的基础 `message` 语义。
- 禁止把原版可选字段改成新的必填项。
- 禁止改变原版 `GET /set_refer_audio?refer_audio_path=...` 的既有调用方式。
- 禁止因为支持桌宠端而替换原版逐次显式传参考字段的工作流。
- 禁止让无扩展字段请求落入魔改逻辑分支。
- 禁止为了低延迟、会话化或流式播放而重写原版默认主线。
- 禁止要求 Yumetsuki 在未显式启用扩展时必须携带 `session_id` 或其他桌宠端专用字段。
- 禁止通过修改原版返回结构来暗中绑定 Yumetsuki 客户端。
- 禁止在扩展失败时移除原版回退路径。
- 禁止把仅供桌宠端使用的扩展字段伪装成原版字段语义变更。
- 魔改只能通过调参、设置项和显式扩展字段实现，不能改动原版逻辑；任何禁止项都不得通过“默认补全”或“兼容优化”绕开。

## Yumetsuki 当前依赖的服务端行为

### 基础依赖

- `/tts` 必须作为统一合成入口。
- `GET /set_refer_audio?refer_audio_path=...` 必须作为当前已支持的参考音频预热路径。
- 原版逐次显式传参考字段的工作流必须保持可用；Yumetsuki 在 `inline` 和 `auto` 回退场景中仍会依赖该路径。
- 服务端必须接受 `text` 与 `text_lang` 组成的基础合成请求；在携带参考字段时，还必须保持对 `ref_audio_path`、`prompt_lang`、`prompt_text` 等显式参考信息的兼容。
- 服务端必须允许句级请求、单句失败跳过和顺序播放语义共存，不得要求整段文本一次性成功才返回可用结果。
- 服务端必须兼容常见语言别名归一化后的结果，例如 `jp -> ja`。

### 扩展能力边界

- `session_preload`、`server_managed` 与 `auto` 属于 Yumetsuki 可利用的兼容策略；服务端若实现相关能力，必须保持原版路径仍可回退。
- `auto` 模式下，若预热失败或服务端仍要求显式参考字段，Yumetsuki 会回退到逐次携带参考字段；服务端不得阻断该回退。
- `session_id`、PCM 流式、结构化错误与元数据头仅属于显式扩展能力；未显式启用时不得影响原版请求。
- 服务端若支持会话能力，只能把 `session_id` 作为可选扩展键，不得把它提升为原版路径前置条件。
- 魔改只能通过调参、设置项和显式扩展字段实现，不能改动原版逻辑；Yumetsuki 当前依赖的所有扩展能力都必须服从这条规则。

## 服务端建议跟进方式（原版路径 / 扩展路径）

### 原版路径

- 无 `session_id`、无其他扩展字段请求时，服务端必须完全按原版路径处理。
- 服务端必须保持 `/tts` 的原版默认行为、原版返回格式与原版错误语义。
- 服务端必须保持逐次显式传参考字段的工作流。
- 服务端必须保持 `GET /set_refer_audio?refer_audio_path=...` 的原版调用方式可用。
- 服务端不得在原版路径中混入桌宠端专用默认补全逻辑。

### 扩展路径

- 仅允许在请求已经显式携带扩展字段时启用扩展路径。
- `session_id` 仅允许作为可选会话键，不得成为原版请求前置条件。
- 会话预热、服务端托管参考、PCM 流式、结构化错误、附加元数据头仅允许作为扩展能力附着在原版路径之上。
- 扩展路径必须保留回退到原版路径的能力；扩展协商失败、预热失败或能力未开启时，不得阻断原版合成。
- 服务端不得把扩展路径抬升为新的默认主线。
- 魔改只能通过调参、设置项和显式扩展字段实现，不能改动原版逻辑；扩展路径必须反复验证这条规则没有被破坏。

## 变更同步要求

- 每次 Yumetsuki 涉及 TTS 对接、兼容策略、请求参数、默认值语义、错误语义、回退逻辑、预热逻辑、响应头或扩展字段的改动，都必须同步更新本文档。
- 每次服务端准备提交与 Yumetsuki 兼容相关的改动时，必须先对照本文档检查是否破坏原版路径。
- 若某项客户端能力被废弃、替换或改语义，必须同步更新本文档，不得只改代码。
- 若服务端新增扩展能力，必须明确标注为显式扩展，不得以“更适合桌宠端”为理由改写原版默认行为。
- 本文档描述的是当前项目对服务端的约束，不是历史草稿；任何过期设计稿都不得替代本文档。
- 魔改只能通过调参、设置项和显式扩展字段实现，不能改动原版逻辑；该规则必须在每次变更同步时再次核对。

## 服务端提交前检查清单

- 原版 `/tts` 默认行为是否未变。
- 原版 `/set_refer_audio` 调用方式是否未变。
- 是否只新增扩展字段而未改旧字段语义。
- 无 `session_id` 时是否仍兼容原版。
- 显式启用扩展时是否仍可回退原版路径。
- 新增错误字段时是否保留原版 `message`。
- `auto`、`inline`、`session_preload`、`server_managed` 的兼容边界是否清晰，且未覆盖原版路径。
- 是否补了原版兼容回归测试。
- 是否同步复核了本文档，确认“魔改只能通过调参、设置项和显式扩展字段实现，不能改动原版逻辑”仍然成立。
