# Tauri 产品 Parity 恢复计划

> 状态：当前 Tauri/Vue 主线不是产品级全量迁移完成态。它只完成了 PySide6 退场、Tauri/Vue 骨架、RPC/sidecar 基础链路和部分自动化 gate。旧版功能、桌宠体验和 UI 样式 parity 仍需系统恢复。

## 问题结论

这次迁移失败的核心原因是验收对象错误：实际验收了技术栈替换和架构 gate，没有验收产品功能、视觉、交互、桌宠体验和旧版行为 parity。

不能再把“PySide6 已删除”“Tauri 能启动”“测试通过”描述为“全量迁移完成”。后续应统一称为：

- 技术状态：Tauri/Vue 主线已接管入口。
- 产品状态：旧版功能和 UI parity 未完成。
- 当前任务：恢复旧版功能、交互和视觉一致性。

## 已知问题

- `pnpm dev` 已修正为 Tauri GUI 入口，`pnpm dev:web` 保留为前端 Web 预览。
- 当前 Tauri 页面与旧版 PySide6 产品形态差异明显。
- 当前测试偏向 RPC、sidecar、安全、发布和基础 smoke，不能证明产品功能完整。
- 当前文档和历史计划曾过早宣称迁移完成，后续必须以本文件和恢复清单为准。
- 真实 API、TTS、STT、OCR、MCP、浏览器自动化和桌宠运行状态仍需要实机验收。

## 失败原因复盘

1. 验收标准错误  
   迁移应验收“旧版功能是否完整复刻”，而不是只看 PySide6 是否退场、Tauri 是否跑通、测试是否通过。

2. 缺少旧版功能矩阵  
   删除旧 UI 前没有建立旧版页面、能力、配置项、状态和交互边界的完整清单，导致新实现缺功能但仍被误判为完成。

3. 文档状态过早收口  
   “迁移完成”的表述影响了后续判断，使工作转向清理和收口，而不是缺口修复。

4. 测试覆盖偏架构  
   自动化测试覆盖了基础入口和协议边界，但没有覆盖视觉 parity、桌宠体验、复杂设置、真实语音链路、插件/MCP 和工具链完整行为。

5. GUI 与 Web 边界未作为硬验收  
   入口一度只启动 Vite Web 页面，说明“用户实际打开的是桌面 GUI”没有作为必须验收项。

6. 重构顺序错误  
   正确顺序应是先保持产品体验和功能，再替换技术栈。实际过程偏向先删除旧实现，再补新骨架。

## 旧版截图获取方案

### 首选：从 git 历史恢复旧版运行环境

1. 查找 PySide6 删除前的提交。
2. 创建只读参考 worktree，例如 `../Yumetsuki-pyside6-reference`。
3. 在旧 worktree 中安装旧依赖并启动 PySide6 GUI。
4. 按页面和状态截图，保存到当前主线的 `docs/reference/pyside6-ui/`。

建议截图范围：

- 桌宠主窗口和立绘状态。
- 聊天窗口：空态、输入中、生成中、长文本、失败、重试、TTS/STT 状态。
- 设置中心：API、角色、记忆、Agent、插件、MCP、日志、系统。
- 日志和诊断页：筛选、详情、导出、错误态。
- 插件、MCP、系统控制、Web 自动化和 OCR 相关页面。
- 托盘、悬浮窗、被动气泡和高风险确认对话框。

### 备选：从历史文档和产物找截图

搜索仓库和历史提交中的 `png`、`jpg`、`webp`、`screenshot`、`smoke`、`ui`、`chat`、`settings` 等关键词。此方式较快，但覆盖通常不完整。

### 兜底：从旧 PySide6 源码重建视觉规格

如果旧程序跑不起来，就从旧源码中恢复窗口尺寸、布局层级、样式表、控件状态、默认文案和交互逻辑，再转写为 UI parity 规格。

## 恢复路线

推荐采用“核心链路优先，功能和视觉并行恢复”的路线：

1. 建立旧版功能矩阵  
   输出 `docs/reference/pyside6-ui/ui-parity-inventory.md`，每一行包含旧版页面/功能、截图、当前 Tauri 状态、缺口、验收方式。

2. 固定验收基线  
   每个页面至少有旧版截图、当前截图、差异说明和验收结论。没有截图时必须有旧源码证据。

3. 优先恢复核心体验  
   顺序为桌宠主窗口、聊天、设置、日志/诊断、插件/MCP/工具、TTS/STT/OCR、系统控制和浏览器自动化。

4. 每项恢复都绑定 GUI 验收  
   不能只用单元测试或 RPC 测试宣布完成。必须至少包含 Tauri GUI 启动、页面可达、关键状态可见和用户路径可操作。

5. 最后再做全局视觉统一  
   在功能可用后，统一 Sakura 主题、间距、字体、窗口尺寸、控件密度、动效和桌宠角色表现。

## 恢复验收标准

单项功能恢复完成必须同时满足：

- 旧版功能矩阵中对应项标记为“已恢复”。
- Tauri GUI 中能完成同等用户路径。
- 样式与旧版截图或已批准的新视觉规格一致。
- 错误、加载、取消、重试、权限和空态有明确表现。
- 至少有聚焦自动化测试或手工 smoke 记录。
- 文档同步更新，不再使用“应当已经完成”的含糊描述。

整体恢复完成必须满足：

- 旧版核心功能矩阵全部闭环。
- 所有核心页面有截图对照记录。
- `pnpm dev` 打开 Tauri GUI，`pnpm dev:web` 仅用于前端预览。
- 真实 API / TTS / STT / OCR / MCP / Playwright Edge 实机验收完成并记录。
- 发布安全、性能预算、RPC contract、no-PySide6 sidecar 和文档 stale gate 继续通过。
