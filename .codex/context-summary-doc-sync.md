### 1. 相似实现分析

- **实现1**: `docs/README.md`
  - 模式：作为主文档入口，只保留当前有效专题和阶段状态。
  - 可复用：当前进度、下一步方向、推荐阅读顺序。
  - 需注意：已完成且并入主文档的历史 spec / plan 不应继续出现在入口。
- **实现2**: `CLAUDE.md`
  - 模式：只保留 AI / 协作者最小上下文，不堆叠完整设计细节。
  - 可复用：当前阶段、关键约束、文档入口。
  - 需注意：页面展示名必须与当前 UI 一致。
- **实现3**: `docs/development.md`
  - 模式：集中维护环境、规划文档、配置、测试策略和 Git 约定。
  - 可复用：规划文档分层、测试入口、配置敏感性说明。
  - 需注意：当前计划只应指向未来 Phase 5 / 6，不再指向已完成日志工作台 plan。
- **实现4**: `docs/ui-guidelines.md`
  - 模式：记录可复用 UI 规范，而非单次修复说明。
  - 可复用：设置中心导航、下拉框、平台日志页面、API 页面控件规则。
  - 需注意：统一下拉框样式应写成全局设置中心规范。

### 2. 项目约定

- **命名约定**: 文档标题和页面名使用当前 UI 文案：`API`、`角色`、`平台日志`。
- **文件组织**: 已完成历史设计从 `docs/superpowers/specs/` 和 `docs/superpowers/plans/` 删除；稳定规范沉淀到 `docs/README.md`、`docs/development.md`、`docs/architecture.md`、`docs/ui-guidelines.md`。
- **导入顺序**: 本次无代码导入调整。
- **代码风格**: 本次文档变更使用现有 Markdown 层级和中文列表风格。

### 3. 可复用组件清单

- `ui/theme.py`: 共享 Sakura 菜单与下拉框主题。
- `ui/assets/combo-down.svg`: 设置中心统一下拉箭头资源。
- `ui/settings/window.py`: 设置中心导航顺序和页面索引绑定。
- `ui/settings/pages/system_log_page.py`: 平台日志页面样式、配色、滚动保持和双视图实现。
- `tests/test_settings_window.py`: 导航、下拉框、API TTS 语言控件回归测试。

### 4. 测试策略

- **测试框架**: pytest。
- **测试模式**: 全量回归 + 相关 UI 文件语法检查 + 文档旧引用扫描。
- **参考文件**: `tests/test_settings_window.py`、`tests/test_system_log_page.py`。
- **覆盖要求**: 导航文案与顺序、共享下拉框样式、API TTS 语言下拉框、平台日志关键缺陷回归、文档旧入口清理。

### 5. 依赖和集成点

- **外部依赖**: PySide6、pytest。
- **内部依赖**: `SettingsWindow` 页面导航、设置页 `QComboBox` 样式、`SystemLogPage` 平台日志展示。
- **集成方式**: 文档同步记录已落地 UI 行为；测试验证现有实现不回退。
- **配置来源**: `data/config/agent.yaml` 仍有用户本地未提交改动，本次提交必须排除。

### 6. 技术选型理由

- **为什么用这个方案**: 已完成的日志工作台 spec / plan 不再作为活跃入口，主文档吸收稳定结论，减少后续新会话误读历史计划。
- **优势**: 文档入口更短，当前页面名与 UI 一致，后续计划焦点转向 Phase 5 / 6。
- **劣势和风险**: 删除历史 spec / plan 后，详细实施过程只能通过 Git 历史追溯；主文档必须保留足够状态总结。

### 7. 关键风险点

- **并发问题**: 本次无运行时代码并发变化。
- **边界条件**: “系统日志”可作为内部 channel / 目录名保留，但页面展示名统一为“平台日志”。
- **性能瓶颈**: 本次无性能路径变化。
- **安全考虑**: 不提交真实 API key，不暂存 `data/config/agent.yaml` 的用户本地调参。
